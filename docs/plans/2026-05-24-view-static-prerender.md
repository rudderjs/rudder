# view() static prerender — Phase 1

**Date:** 2026-05-24
**Author:** Suleiman Shahbari

## Summary

Add opt-in build-time static rendering for `view('id')` routes. Marketing pages, docs index, terms / privacy / 404 — anything whose markup doesn't depend on a per-request state — gets served as plain HTML from disk instead of paying SSR cost on every hit.

**Mechanism:** `export const prerender = true` at the top of `app/Views/Foo.tsx`. The view scanner picks this up the same way it picks up `export const route = '...'` and emits a Vike `+prerender.ts` next to the generated `+Page.tsx`. `vike build` then emits `dist/client/<url>.html` and the production server serves the static file before falling back to SSR.

**Phase 1 scope:** static URL only (no `:param` segments). Dynamic prerender (`/blog/:slug` → enumerate slugs and emit one HTML per slug) is a Phase 2 follow-up — needs an `onBeforePrerenderStart` hook the user supplies.

## Why this design

- **Opt-in, not auto-detect.** Auto-detection would have to infer "is this view safe to prerender?" — false positives (a view that LOOKS pure but reads from `pageContext`) ship broken HTML. Opt-in is cheap and explicit.
- **Marker in the view file**, not the controller. The view is where "this is pure" is true. Controllers may still exist for prerendered views (the build calls the controller once at prerender time to seed default props), but the gate lives at the rendering surface.
- **Scanner regex, not AST.** Same pattern as `ROUTE_EXPORT_RE` / `PROPS_EXPORT_RE` in `packages/vite/src/views-scanner.ts` — no new TS-parser dep, multiline-tolerant for `.vue` SFCs.
- **Vike does the heavy lifting.** Vike has a mature `+prerender` / `+config: { prerender: true }` pipeline. We don't reinvent it — we just opt our virtual pages in.

## File-level changes

### `packages/vite/src/views-scanner.ts`

- New regex `PRERENDER_EXPORT_RE = /(?:^|[\s;])export\s+const\s+prerender\s*(?::\s*boolean)?\s*=\s*true\b/m`
- New `readPrerenderOpt(absPath): boolean` mirroring `readRouteOverride`
- `DiscoveredView` gains `prerender: boolean`
- In the leaf `__view/<id>/` output, emit a new `+prerender.ts`:
  ```ts
  // AUTO-GENERATED by @rudderjs/vite — do not edit.
  export default true
  ```
  Only when `prerender === true`. When false (default), no file is emitted — keeps the per-view dir clean for unaffected views.

### `packages/view/src/index.ts`

- No runtime change. `view('id', props)` still returns the same `ViewResponse` for SSR fallback (prerender doesn't change request-time behaviour for non-prerendered views, and for prerendered views the production server serves the HTML before the request ever hits the controller — so controllers for prerendered views are dead code in prod).
- Doc the gotcha in `ViewOptions` JSDoc: when a view is prerendered, the controller still runs in dev (where prerender is a no-op) and at build time (once, to seed any default props the prerender step needs). Keep controllers idempotent and side-effect-free for prerendered views.

### `packages/view/src/types/vike.ts`

- No change. Vike's own types cover `+prerender`.

### Generated `pages/__view/<id>/+prerender.ts` (new)

- Single-line `export default true`.

### Documentation

- `docs/guide/views.md` — new "Prerendering" section: when to opt in, the gotcha about dynamic data, link to Vike's prerender docs for advanced cases.
- `packages/view/README.md` — single-paragraph mention of the flag.
- `CLAUDE.md` — Common Pitfalls entry on "controller runs at build time for prerendered views; keep it pure."

- Verify `pnpm build` emits `playground/dist/client/index.html` (or wherever the welcome route lives — `/` per its current `export const route = '/'`).

## What does NOT change

- **Dev mode** — Vike's prerender is build-time only. `pnpm dev` still SSRs every request, prerender flag is inert.
- **`view()` API** — `view('id', props)` works the same. Apps that don't opt anything in are exactly unchanged.
- **Controllers** — Controllers for prerendered views still get registered with the router; just the production runtime never invokes them because the static file is served first. (Vike's runtime handles this.)
- **Auth-guarded views** — Trivially incompatible with prerender (the user is unknown at build time). The opt-in mechanism + per-view marker means apps can't accidentally prerender `Dashboard.tsx`.

## Phase 2 follow-up (out of scope here)

- Dynamic prerender: `export const prerender = () => fetchAllSlugs()` returning a list of URLs. Scanner emits `+onBeforePrerenderStart.ts` instead of `+prerender.ts`.
- Per-URL props for the dynamic case (e.g. blog post titles for `<title>` tags) — needs a controller-at-prerender-time path.
- Smarter cache headers in `server-hono` for the served static files (`Cache-Control: public, max-age=31536000, immutable` for content-hashed assets, shorter for HTML).

## Open questions

1. **Does Vike's `+prerender: true` work with our string-default `+route.ts`?** Should — Vike's docs say string routes are valid for prerender. Verify during impl via the playground smoke.
2. **Build CLI integration.** `pnpm rudder build` is currently a thin wrapper around `vike build`. The prerender output should be emitted automatically. Verify; if not, add an `--prerender` flag the rudder CLI passes through. **Likely no change needed** since prerender is per-page opt-in.
3. **Vue / Solid scanner stubs.** The scanner emits framework-specific `+Page.tsx` / `+Page.vue` / `+Page.jsx` files; the `+prerender.ts` is framework-agnostic. Should work uniformly across frameworks — verify with vue/solid scaffolder smoke.
