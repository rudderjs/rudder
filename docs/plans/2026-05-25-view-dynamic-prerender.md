# view() dynamic prerender — Phase 2

**Date:** 2026-05-25
**Author:** Suleiman Shahbari
**Builds on:** `docs/plans/2026-05-24-view-static-prerender.md` (Phase 1, shipped via #616)

## Summary

Extend the view scanner so `export const prerender = [...] | () => [...]` enumerates URLs to prerender at build time for parameterized view routes — blog posts, docs articles, anything with a `/:slug`-style segment.

```ts
// app/Views/Blog/Post.tsx
export const route = '/blog/@slug'
export const prerender = ['/blog/hello-world', '/blog/another-post']

export default function Post({ slug }: { slug: string }) {
  return <article>{slug}</article>
}
```

`pnpm build` emits `dist/client/blog/hello-world/index.html` and `dist/client/blog/another-post/index.html`. The function form (`async () => fetchSlugs()`) lets the URL list come from a DB query, filesystem walk, or remote API — whatever the user wires.

## Why this design

- **Vike already has the right hook.** `+onBeforePrerenderStart` is Vike's per-page mechanism for parameterized prerender. We don't invent — we just re-export the user's `prerender` symbol from it.
- **Same surface as Phase 1.** Static (`= true`) and dynamic (`= [...]` / `= () => ...`) live on the **same exported name** in the same view file. Authors don't learn a second concept; the scanner picks the right output based on the RHS shape.
- **Scanner regex, not AST.** Continues the file's existing detection style (`ROUTE_EXPORT_RE`, `PROPS_EXPORT_RE`, `PRERENDER_EXPORT_RE`). Multiline-tolerant for Vue SFCs out of the box.
- **Pass-through return shape.** Whatever the user's `prerender` returns (string array or `{ url, pageContext }` array, sync or async) gets handed straight to Vike. No reinterpretation, no coercion — Vike's `OnBeforePrerenderStartSync<unknown>` / `OnBeforePrerenderStartAsync<unknown>` already cover all the shapes that matter.

## File-level changes

### `packages/vite/src/views-scanner.ts`

- Replace the single `prerender: boolean` field on `DiscoveredView` with `prerender: 'off' | 'static' | 'dynamic'`.
- New regex `PRERENDER_DYNAMIC_RE` matches `= [`, `= (`, `= async (`, `= function`, `= async function` (with optional `: type` annotation before the `=`).
- `readPrerenderOpt(absPath)` now returns the mode discriminator. Static beats dynamic when both match somehow (defensive — only matters for pathological source).
- New `+onBeforePrerenderStart.ts` codegen, emitted when `mode === 'dynamic'`. The generated file imports `prerender` (named export) from the view file and exposes it as Vike's hook. Both sync-array and async-function returns flow through unchanged:

  ```ts
  // AUTO-GENERATED by @rudderjs/vite — do not edit.
  // Source: `export const prerender = [...] | () => ...` in App/Views/<file>
  import type { OnBeforePrerenderStartAsync } from 'vike/types'
  import { prerender as source } from 'App/Views/Blog/Post.tsx'

  export const onBeforePrerenderStart: OnBeforePrerenderStartAsync<unknown> =
    async () => (typeof source === 'function' ? await source() : source)
  ```

  Always async-typed — sync arrays and sync functions also work because `() => Promise<X[]>` accepts a function that resolves to `X[]`.
- Vike requires **both** `+prerender: true` AND `+onBeforePrerenderStart` for parameterized prerender to actually run. The scanner emits both files when `mode === 'dynamic'`.
- Cleanup is symmetric: switching from dynamic → off removes both files; static → dynamic removes the `+prerender.ts` (still re-emitted) and adds the hook file; etc. Centralize via a `prerenderArtifacts(mode)` helper rather than duplicating the conditional logic.

## Detection rules

The regex pair, declared together so the discriminator logic is easy to read:

```ts
const PRERENDER_STATIC_RE  = /(?:^|[\s;])export\s+const\s+prerender\s*(?::\s*boolean)?\s*=\s*true\b/m
const PRERENDER_DYNAMIC_RE = /(?:^|[\s;])export\s+const\s+prerender(?:\s*:[^=]+)?\s*=\s*(?:\[|\(|async\s+(?:\(|function)|function)\b/m

function readPrerenderOpt(absPath: string): 'off' | 'static' | 'dynamic' {
  const text = fs.readFileSync(absPath, 'utf8')
  if (PRERENDER_STATIC_RE.test(text))  return 'static'   // `= true` wins ties
  if (PRERENDER_DYNAMIC_RE.test(text)) return 'dynamic'
  return 'off'
}
```

Matched shapes (mentally verified):

| Source | Mode |
|---|---|
| `export const prerender = true` | static |
| `export const prerender: boolean = true` | static |
| `export const prerender = false` | off (Vike's `+prerender: false` is the default; emitting the file would be wrong) |
| `export const prerender = []` | dynamic — empty array still opts in (Vike treats it as "no URLs for this page") |
| `export const prerender = ['/a', '/b']` | dynamic |
| `export const prerender = () => [...]` | dynamic |
| `export const prerender = async () => [...]` | dynamic |
| `export const prerender = function() { ... }` | dynamic |
| `export const prerender = async function() { ... }` | dynamic |
| `export const prerender: string[] = [...]` | dynamic |
| `export const prerender: () => Promise<string[]> = async () => [...]` | dynamic |
| `export const prerender = SOME_CONST` | off (can't statically determine — fail closed; opt-in is explicit) |

Variable-reference RHS is intentionally not picked up. Authors who want dynamic enumeration via a const can inline it: `export const prerender = MY_LIST.slice()` matches the array-literal-or-call shape regex via... actually no, `MY_LIST.slice()` matches `(` in `slice(`. Hmm — that's a false positive. Mitigation: tighten the regex so `(` only matches when preceded by `=` (arrow or paren-call), not `.`. The regex above already requires `=\s*(` so this is fine — `MY_LIST.slice()` has `=` separated from `(` by `MY_LIST.slice`, so it doesn't match. Good.

## What does NOT change

- **Static prerender path** — `export const prerender = true` still emits exactly the same `+prerender.ts` it did in Phase 1. Phase 2 is purely additive.
- **Dev mode** — still SSR every request. Prerender hooks are build-time only.
- **`view()` API** — no runtime change.
- **Phase 1 cleanup logic** — extended to delete `+onBeforePrerenderStart.ts` alongside `+prerender.ts` when the export is removed.
- **No controller-at-build-time path** — Phase 1 noted "the controller still runs at build time for prerendered views to seed default props," but for the dynamic case, the controller is **not** called per-URL at build time. Vike passes route params via `pageContext.routeParams` to the page component. View props that need per-URL data must come back from the user's `prerender` function as the `pageContext` field of `{ url, pageContext }` entries.

## Phase 3 follow-up (out of scope)

- **Controller invocation at build time per URL.** Mirrors Laravel's "request-scoped" model — let the controller receive route params and return view props at build time, so `view('blog.post', { post })` works for prerendered URLs without the user wiring `pageContext` themselves. Needs a build-time HTTP shim and a way to enumerate URLs separately from rendering them.
- **`prerender: 'auto'` mode** that infers URLs from a `findManyForPrerender()` static on a model. Convention-over-config; lower priority until we see real usage.
- **Cache-Control headers** for served prerendered files in `server-hono`.

## Regression tests

- `packages/vite/src/views-scanner.test.ts`:
  - `prerender = ['/a', '/b']` → `+prerender.ts` AND `+onBeforePrerenderStart.ts` both emitted; the latter contains `import { prerender as source } from 'App/Views/...'`.
  - `prerender = async () => [...]` → same (both files emitted).
  - Removing the export → both files cleaned up.
  - Switching from static (`= true`) to dynamic (`= [...]`) → `+prerender.ts` unchanged on disk (idempotent write), `+onBeforePrerenderStart.ts` added.
  - Switching from dynamic to static → `+onBeforePrerenderStart.ts` removed, `+prerender.ts` stays.
  - False-positive guard: `export const prerender = SOMETHING.slice()` (variable reference + call) is `off`. (Today's static regex doesn't match this; the dynamic regex requires `=\s*(` which doesn't either — the test pins this.)
