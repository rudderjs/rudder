# DX Quick-Wins Roadmap

**Status:** roadmap, 2026-05-21. Three small-to-medium DX phases to ship after the brand sweep + create-rudder rename + installer logo landed.
**Origin:** post-rebrand DX audit (session 2026-05-21). Pitched typed bodies; turned out shipped via #564. Re-greppped the surface for actual gaps and landed on these three.

---

## TL;DR

| # | Phase | Effort | Risk | Order |
|---|---|---|---|---|
| 1 | **VS Code `.vscode/` scaffolder templates** | S (half day) | low | first |
| 2 | **Missing `make:*` generators** (policy / gate / notification / resource) | M (~1 day) | low | second |
| 3 | **Shell completions** (`rudder <tab>`) | M (1–2 days) | medium | third |

Total ~3–4 days of work, three separate PRs, each independently shippable.

## Sequencing rationale

Easiest first to build momentum, biggest-swing last when patterns are warm:

- **VS Code first** — self-contained scaffolder template addition. Zero framework changes. Ships in an afternoon and every new `npm create rudder` benefits.
- **`make:*` second** — extends a well-established pattern (`make:request` is the template). Mostly typing, no architecture decisions. One day batched.
- **Completions last** — most architectural decisions (lib choice, rc-file behavior, install/uninstall flow). Worth doing after the cheap wins are banked.

## Phase 1 — VS Code `.vscode/` templates

### Goal

`npm create rudder@latest my-app` scaffolds a project that is **F5-debuggable in VS Code / Cursor out of the box** — no manual launch.json authoring.

### What ships

Three files added to the scaffold output, generated by `create-rudder-app/src/templates.ts`:

**`.vscode/launch.json`** — three Node debug configurations:

1. **"Debug dev server"** — spawns the dev command with `--inspect-brk`, attaches the debugger. Sourcemaps wired so breakpoints work in `routes/`, `app/`, and `bootstrap/`.
2. **"Debug rudder command"** — input prompt for command name, debugs `tsx node_modules/@rudderjs/cli/src/index.ts <cmd>`. Lets users F5 into `make:*`, `db:push`, `tinker`, etc.
3. **"Debug current test file"** — runs `node --test ${file}` with `--inspect-brk`. Active editor file is the test target.

**`.vscode/extensions.json`** — recommended-extensions list, populated based on scaffolder choices:
- Always: Vite, Tailwind CSS IntelliSense
- Conditional: Prisma (if ORM = Prisma), Vue (if framework includes Vue)

**`.vscode/settings.json`** — minimal, opinionated defaults:
- `typescript.tsdk: node_modules/typescript/lib` — uses workspace TS, not VS Code's bundled one
- `editor.formatOnSave: false` — don't enforce a formatter the user hasn't chosen

### Scope decisions

- **Always commit `.vscode/`** (not gitignored). The configs describe project debug entry points, not personal preferences — same category as `package.json` scripts. Users on other editors carry a small unused directory; deletable with one `rm -rf .vscode`.
- **No scaffolder prompt** ("Include VS Code config? Y/n"). Just always emit. The cost of a wrong prompt answer is low; the cost of a missing flag is users who never know it was an option.
- **Cursor compatibility free.** Cursor is a VS Code fork, reads the same files. No separate `.cursor/` directory needed.

### Files to touch

- `create-rudder-app/src/templates/vscode.ts` — new module, exports `launchJson()`, `extensionsJson(answers)`, `settingsJson()`
- `create-rudder-app/src/templates.ts` — wire into the templates pipeline alongside the existing config emitters
- `create-rudder-app/src/templates.test.ts` — snapshot tests for the three files
- `create-rudder-app/scripts/smoke.ts` — assert `.vscode/launch.json` exists after a default scaffold

### Out of scope

- Webstorm `.idea/` config — different audience, different format, defer
- Vim/Neovim DAP config — niche, deferrable
- Editor-agnostic `dev.sh` debug helper — solving a problem nobody has

### Verification

- `pnpm --filter create-rudder-app build && pnpm --filter create-rudder-app test`
- Fresh scaffold, open in VS Code, press F5 in `routes/web.ts` → breakpoint hits on first request
- Snapshot tests catch any unintended drift in the three emitted files

### PR scope

One PR. Changeset: minor bump on `create-rudder-app` (new scaffolded files).

## Phase 2 — Missing `make:*` generators

### Goal

Four common Laravel-equivalent scaffolders that are missing from the 18 already-shipped: `policy`, `gate`, `notification`, `resource`. Each follows the `make:request` pattern exactly — zero architecture, mostly typing.

### What ships

Four new make commands, registered the same way the existing ones are:

**`rudder make:policy <Name>`** — generates `app/Policies/<Name>Policy.ts` with the Eloquent-style policy shape (`view`, `create`, `update`, `delete` methods receiving `User` and the target model). Owned by `@rudderjs/auth`'s commands subpath since policies belong to the auth domain.

**`rudder make:gate <Name>`** — generates `app/Gates/<Name>Gate.ts` stub registering a single named gate via `Gate::define()`. Also owned by `@rudderjs/auth`.

**`rudder make:notification <Name>`** — generates `app/Notifications/<Name>Notification.ts` with the channel-router shape (`via()` method returning array of channels, `toMail()`, `toDatabase()` implementations stubbed). Owned by `@rudderjs/notification`.

**`rudder make:resource <Name>`** — generates `app/Http/Resources/<Name>Resource.ts` — API resource transformer (Laravel-style). Takes a model, returns a JSON shape. Owned by `@rudderjs/http`.

### Pattern (already established)

Look at `packages/cli/src/commands/make/request.ts` for the canonical shape. Each new generator is ~40 lines: a `MakeSpec` object declaring path / template / replacements, registered via `registerMakeSpecs()` from `@rudderjs/console`.

### Scope decisions

- **Owned by domain packages, not CLI** — keeps the CLI runner thin (per `packages/cli/CLAUDE.md` architecture rule). `@rudderjs/auth` exports `policy` + `gate` from a `commands/` subpath; `@rudderjs/notification` exports `notification`; `@rudderjs/http` exports `resource`. The CLI's `loadPackageCommands()` already eagerly imports known subpaths — add the four new ones to that list.
- **Template fidelity** — generated files compile and pass typecheck out of the box. No `// TODO: implement` placeholders that break the build.
- **No interactive prompts.** All four follow the existing `make:*` style — one positional argument, generates the file. `--resource` flag on `make:policy` to scaffold all 7 methods (Laravel parity) is a nice-to-have, defer.

### Files to touch

- `packages/auth/src/commands/make-policy.ts` (new)
- `packages/auth/src/commands/make-gate.ts` (new)
- `packages/auth/package.json` — add subpath exports
- `packages/notification/src/commands/make-notification.ts` (new)
- `packages/notification/package.json` — add subpath export
- `packages/http/src/commands/make-resource.ts` (new)
- `packages/http/package.json` — add subpath export
- `packages/cli/src/index.ts` — add the four loader entries to `loadPackageCommands()` (per CLAUDE.md: "If you add a new package command, add the loader entry in `packages/cli/src/index.ts`")
- Tests next to each new make file (template-emit snapshot, mirror the existing patterns)

### Out of scope (defer)

- `make:scope`, `make:cast`, `make:observer` — niche, can come later
- `make:test` — tests usually live next to source, not in a separate `tests/` tree
- `--resource` flag on `make:policy` (Laravel-style all-7-methods)

### Verification

- `pnpm --filter @rudderjs/auth typecheck && pnpm --filter @rudderjs/auth test`
- Same for notification + http
- From `playground/`: `pnpm rudder make:policy Post && pnpm rudder make:gate IsAdmin && pnpm rudder make:notification Welcome && pnpm rudder make:resource User` — verify each file lands at the expected path and compiles

### PR scope

One PR, four packages bumped. Each domain package gets a `minor` changeset (new command surface).

## Phase 3 — Shell completions

### Goal

`rudder <TAB>` works in bash, zsh, and fish. Including:
- `rudder <TAB>` → lists all built-in + package-contributed commands
- `rudder make:<TAB>` → narrows to the ~22 generators
- `rudder migrate:<TAB>` → suggests `:fresh`, `:status`
- `rudder make:m<TAB>` → narrows to `make:mail`, `make:middleware`, `make:migration`, `make:model`, `make:mcp-*`

### What ships

Three new rudder subcommands and one library dep:

**`rudder completion <shell>`** — emits a raw completion script for `bash` / `zsh` / `fish` to stdout. For users who manage their own dotfiles (`rudder completion zsh > ~/.zsh/completions/_rudder`).

**`rudder completion install`** — interactive: detects current shell, writes a sourced file to a stable location (`~/.rudder/completion.<shell>`), appends a single `source ~/.rudder/completion.<shell>` line to the user's rc file (`~/.zshrc`, `~/.bashrc`, or fish's completions directory). Idempotent — re-running is a no-op.

**`rudder completion uninstall`** — removes the `source` line and the sourced file. Clean exit.

**Library:** `tabtab` (npm, ~1k stars, handles all 3 shells). Chosen over Commander's built-in completion because Commander only supports bash; `tabtab` covers bash + zsh + fish from a single source.

### Discovery flow

Two surfaces nudge users toward `rudder completion install`:

1. **`create-rudder` post-scaffold panel** — one-line hint: `Shell completions: pnpm rudder completion install` appended to the `Done!` panel's hints array (alongside the existing AI / MCP / Telescope hints).
2. **`rudder --help` footer** — single line at the bottom: `Tip: enable tab-completion with \`rudder completion install\``

No prompt during scaffold itself (avoids prompt fatigue).

### Scope decisions

- **Static completions only in v1.** Phase 1 lists known command names. Dynamic completions (`rudder model:show <TAB>` → model names from `app/Models/`) is a separate phase — small to add once the static path lands, but real scope creep if bundled.
- **`~/.rudder/completion.<shell>`** as the sourced file location. Not `~/.config/rudder/` — keeps the rudder CLI's footprint discoverable (`ls ~/.rudder` shows what we added). User can delete the whole directory to cleanly uninstall.
- **Idempotent install** — detect existing source line, skip re-add. Cleanly handles `completion install` run twice.
- **Bash + zsh + fish only.** PowerShell / Nushell deferred — small fraction of users and require entirely different syntax.

### Files to touch

- `packages/cli/src/commands/completion.ts` (new) — three subcommands (`<shell>`, `install`, `uninstall`)
- `packages/cli/src/index.ts` — register the new command, add the `--help` footer line
- `packages/cli/package.json` — add `tabtab` to `dependencies`
- `create-rudder-app/src/index.ts` — add the post-scaffold hint line
- `packages/cli/src/commands/completion.test.ts` — tests for script emission + install/uninstall idempotency (mock fs)

### Out of scope (defer to v2)

- Dynamic completions (model names, route names, migration names)
- PowerShell / Nushell scripts
- Per-flag completions inside commands (`rudder make:model --connection=<TAB>` listing connections)
- An `npm postinstall` auto-install — too invasive; users opt in explicitly

### Risk + mitigation

- **rc-file edits are invasive.** Mitigated by writing only one `source` line + storing the actual completion in `~/.rudder/`. Uninstall removes both cleanly.
- **`tabtab` is a single point of failure.** It's stable and small. If we ever outgrow it, the shell scripts it emits are well-documented and we can hand-roll a replacement without changing the user-visible interface.
- **Cross-shell behavior differences** — `bash` and `zsh` mostly share, `fish` is its own world. `tabtab` handles the differences; verification needs a real test in each of the three shells before PR.

### Verification

- `pnpm --filter @rudderjs/cli build && pnpm --filter @rudderjs/cli test`
- Manually in a fresh terminal:
  - `pnpm rudder completion install` (each shell)
  - Open a new shell session
  - `rudder <TAB>` → completion lists all commands
  - `rudder completion uninstall` → confirm rc file restored, `~/.rudder/` cleaned

### PR scope

One PR. Changeset: `minor` bump on `@rudderjs/cli` (new command surface). `create-rudder-app` gets a `patch` for the hint line.

## What this doesn't include

Things that came up during the audit but are deliberately deferred:

- **`make:scope`, `make:cast`, `make:observer`, `make:test`** — niche generators, hold for explicit user demand
- **Did-you-mean / typo error hints** on unknown rudder commands — small effort but mostly noise (Commander already shows "unknown command X" with suggestions). Revisit if users complain.
- **`route:list` polish** — the 136-line implementation is fine; no specific user pain.
- **Inline tutorials / post-scaffold guided tour** — bigger swing, harder to scope, hold for the AI-assisted scaffolding plan.
- **AI-assisted scaffolding** (`rudder ai-generate model Post with author belongsTo user`) — separate plan doc when there's appetite for it.

## Order of merge

```
Phase 1  (VS Code config)      → PR A → publish create-rudder-app minor
Phase 2  (make:* generators)   → PR B → publish auth + notification + http minors

```

Each PR is independently mergeable + revertable. No cross-phase dependencies.

Suggested cadence: Phase 1 same-day, Phase 2 next session, Phase 3 the following session when there's room for the cross-shell verification.
