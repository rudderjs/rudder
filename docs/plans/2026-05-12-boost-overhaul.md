# Boost overhaul — structure + skill modularization

**Status:** planning, 2026-05-12.
**Effort:** ~1–2 days, single PR.
**Scope:** `@rudderjs/boost` only — no changes to other packages.
**Next PR (out of scope here):** coverage push — author guidelines for the 10 user-facing packages currently missing them.

## Why

We ran `php artisan boost:install` on a Laravel 13 project and compared the generated artifacts head-to-head against our `pnpm rudder boost:install` output. Concrete deltas:

| Aspect | Laravel | RudderJS today |
|---|---|---|
| CLAUDE.md length | 212 lines, structured | **1,353 lines, mostly duplicated** content from `.ai/guidelines/*.md` |
| CLAUDE.md structure | XML wrapper `<laravel-boost-guidelines>` + `=== <name> rules ===` dividers + ecosystem-context table + Skills Activation section with explicit `TRIGGER when:` / `SKIP:` heuristics | Plain markdown with `---` HR dividers, no foundational context, no activation logic |
| Skill shape | Largest skill (`laravel-best-practices`) is `SKILL.md` + `rules/` subdirectory with 20 modular files | Every skill is one monolithic `SKILL.md`, 230–285 lines each |
| Skill frontmatter | `name`, `description`, `license`, `metadata.author` | `name`, `description` only |
| Pattern shape | Skills use paired Incorrect / Correct code blocks consistently | Only positive examples + a "Common pitfalls" prose list |

The 1.3k-line CLAUDE.md is the highest-leverage fix: it burns context window in every project using `@rudderjs/boost`, on every AI request, forever — and most of it is verbatim duplication of files that already live in `.ai/guidelines/`.

## What ships

| Component | Path |
|---|---|
| Compact `CLAUDE.md` generator (XML wrapper, dividers, foundational context, activation section, package pointers — not content dumps) | `packages/boost/src/agents/claude-code.ts` (or wherever the generator lives) |
| Skill frontmatter — add `license` + `appliesTo` | `packages/boost/src/templates/skills/*.md` (or skill source files) |
| Skill activation registry — explicit `trigger` / `skip` metadata used to populate the CLAUDE.md Skills Activation section | `packages/boost/src/agents/claude-code.ts` |
| Modularize 3 largest skills into `SKILL.md + rules/*.md` | `packages/boost/src/templates/skills/{orm-models,auth-setup,mcp-servers}/` |
| Adopt paired Incorrect / Correct pattern in the new rule files | (skill content) |
| Regenerate `playground/CLAUDE.md` + `playground/.ai/` as proof | `playground/` |
| Changeset (`@rudderjs/boost` minor — output shape change, no API break) | `.changeset/` |

## Out of scope (next PR)

The **coverage push** — author `boost/guidelines.md` for the ~10 user-facing packages still missing them (`cli`, `http`, `middleware`, `hash`, `mail`, `notification`, `cache`, `session`, `passport`, `view`). Deliberately split because:

- different review surface (doc-heavy vs format/code)
- we want the new format locked in before writing 10 more guidelines into it
- per `project_boost_guidelines_not_published.md`, also needs `"boost"` added to `package.json` `files` field — bundled with this same coverage PR

## Detail — new CLAUDE.md shape

Target ≤250 lines. Template:

```
<rudderjs-boost-guidelines>

=== foundation rules ===

# RudderJS Boost Guidelines

These guidelines are generated by `@rudderjs/boost` from your installed
`@rudderjs/*` packages. Treat them as the project's source of truth for
framework patterns.

## Foundational Context

- node — {{ node_version }}
- @rudderjs/core — {{ pkg_version }}
- @rudderjs/orm — {{ pkg_version }}
- @rudderjs/ai — {{ pkg_version }}
- … (one line per installed @rudderjs/* package)

## Skills Activation

Skills live in `.ai/skills/`. Activate them based on file context.

- `orm-models` — ACTIVATE when: editing a file under `app/Models/` or
  importing from `@rudderjs/orm`. SKIP when: editing a route handler that
  only consumes a model — load `controller-views` instead.
- `auth-setup` — ACTIVATE when: configuring guards, gates, or vendoring
  auth views. SKIP when: writing a route that just calls `Auth::user()` —
  no skill needed.
- … (one line per installed skill)

=== boost rules ===

# Boost MCP Tools

When working in this project, prefer Boost MCP tools over re-deriving:

- `route_list` — current HTTP routes (don't grep `routes/`)
- `model_list` — installed ORM models
- `app_info` — installed @rudderjs/* versions
- `db_schema` — current Prisma schema
- `search_docs` — query installed @rudderjs/* package docs
- … (one line per tool)

=== per-package guidelines ===

For per-package patterns, load the matching file:

- `@rudderjs/ai` → `.ai/guidelines/ai.md`
- `@rudderjs/auth` → `.ai/guidelines/auth.md`
- … (pointer per installed package, not content)

</rudderjs-boost-guidelines>
```

Everything beyond the pointers stays in `.ai/guidelines/*.md`, untouched.
The agent loads the pointer file on demand instead of carrying 1.3k lines
of guidelines in its base context.

## Detail — skill modularization

Three skills get a `rules/` subdirectory. The split keys off the natural
section boundaries already present in the monolithic files:

**`orm-models/`** (currently 267 lines)
```
SKILL.md                        # 80 lines: when-to-use + Quick Reference table linking to rules
rules/
  defining-models.md            # static table, fillable, casts, soft deletes
  querying.md                   # static query methods, scopes, pagination
  relationships.md              # with(), withWhereHas, polymorphic
  factories.md                  # ModelFactory, sequence, states
  resources.md                  # JsonResource, collection responses
  observers.md                  # Model.observe, lifecycle hooks
```

**`auth-setup/`** (currently 273 lines) — split by lifecycle phase:
```
SKILL.md
rules/
  provider-setup.md             # hash + auth provider order, config/auth.ts
  guards.md                     # SessionGuard, RequireAuth, RequireBearer
  password-reset.md             # PasswordBroker, token repositories
  email-verification.md         # MustVerifyEmail, signed URLs
  gates-and-policies.md         # Gate.define, Policy classes
```

**`mcp-servers/`** (currently 285 lines) — split by concern:
```
SKILL.md
rules/
  defining-server.md            # @McpServer decorator, server config
  defining-tools.md             # @Tool, parameter schemas
  streaming-tools.md            # async function* + modelOutput()
  authentication.md             # bearer/oauth flows for MCP
  observers.md                  # mcp observer registry pattern
```

Smaller skills (`ai-agents`, `ai-tools`, `controller-views`) stay as
single `SKILL.md` files. Modularization is mid-sized-skill medicine, not
universal.

Each new rule file gets:
- One Incorrect / Correct paired code block per major rule
- Cross-link back to the SKILL.md quick reference
- No frontmatter (rule files are loaded only via the parent skill)

## Detail — skill frontmatter

Today:
```yaml
---
name: orm-models
description: Creating Eloquent-style models, queries, ...
---
```

Target:
```yaml
---
name: orm-models
description: Creating Eloquent-style models, queries, relationships, casts, factories, and API resources in RudderJS
license: MIT
appliesTo:
  - '@rudderjs/orm'
  - '@rudderjs/orm-prisma'
metadata:
  author: rudderjs
---
```

`appliesTo` is the new field. It lets Boost suppress skills whose
packages aren't installed (e.g. don't ship `orm-models` if the project
doesn't have `@rudderjs/orm`).

## Detail — activation heuristics

Source-of-truth for activation lives in the skill source file (in
`@rudderjs/boost`'s template directory), so authors keep heuristics next
to the skill content:

```ts
// packages/boost/src/skills.ts
export const skills = [
  {
    name: 'orm-models',
    trigger: 'editing a file under `app/Models/` or importing from `@rudderjs/orm`',
    skip:    'editing a route handler that only consumes a model — load `controller-views` instead',
  },
  // ...
]
```

The generator templates these into the CLAUDE.md Skills Activation
section verbatim.

## Risk + migration

- **Existing apps regenerate** — anyone running `pnpm rudder boost:update` after the release gets the new shape. We document this in the changeset body so users can git-diff and review before committing.
- **Custom `CLAUDE.md` edits get clobbered** — same as today; the generator owns the file. If users have written custom rules they want preserved, they should add them to a separate file outside Boost's territory (we recommend a `CLAUDE.local.md`-style escape hatch in the docs — no code change needed, just doc guidance).
- **`appliesTo` filtering can hide skills** users actually want — bail-out via a new flag `--include-all-skills` on `boost:install` for the edge case. Cheap to add.
