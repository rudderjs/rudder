# Plan: Auto Provider Discovery (Laravel-style)

## Context

The current `bootstrap/providers.ts` requires users to manually import and register every framework package they use. The playground's file is 60+ lines of repetitive boilerplate (`cache(configs.cache)`, `mail(configs.mail)`, etc.) that almost every RudderJS app will need.

Laravel 11's equivalent is one line:
```php
return [App\Providers\AppServiceProvider::class];
```

Framework providers are auto-discovered from `composer.json`'s `extra.laravel.providers` array. Package authors declare themselves once; users never edit `bootstrap/providers.php` for framework packages.

This plan brings the same DX to RudderJS using `package.json` metadata + a build-time manifest.

---

## Goals

1. **Reduce `bootstrap/providers.ts` to ~7 lines** — only app-specific providers (events, AppServiceProvider)
2. **New packages auto-register** — install `@rudderjs/X`, run discover, it boots automatically
3. **Zero magic at runtime** — manifest is generated at build time, runtime just reads JSON
4. **Backward compatible** — existing manual provider lists still work
5. **Enforced boot order** — packages declare dependencies, framework sorts them

---

## Non-Goals

- **Not** scanning `node_modules` at runtime on every cold start (too slow)
- **Not** auto-registering arbitrary npm packages — only those that opt in via `rudderjs` field
- **Not** removing the ability to manually register providers — explicit always wins

---

## Architecture

### 1. Package Self-Declaration

Every `@rudderjs/*` package adds a `rudderjs` field to its `package.json`:

```json
{
  "name": "@rudderjs/cache",
  "rudderjs": {
    "provider": "cache",
    "config": "cache",
    "stage": "infrastructure",
    "depends": ["@rudderjs/log"]
  }
}
```

**Field reference:**

| Field | Required | Type | Description |
|---|---|---|---|
| `provider` | Yes | string | Named export from main entry that returns the ServiceProvider class. E.g. `cache` for `export function cache(config) {...}` |
| `config` | No | string | Key in `configs` object passed to the factory. Omit if no config needed (e.g. `notifications()`) |
| `stage` | Yes | enum | Boot stage — see ordering below |
| `depends` | No | string[] | Package names that must boot before this one |
| `optional` | No | boolean | Skip without error if peer dependencies missing (default: false) |

**Stage values (boot order):**

1. `foundation` — log, contracts, support (no dependencies)
2. `infrastructure` — database, session, hash, cache, auth (depends on foundation)
3. `feature` — queue, mail, storage, schedule, notifications, broadcasting, live, ai, boost, localization
4. `monitoring` — telescope, pulse, horizon (depends on infrastructure + features they observe)
5. `application` — user code (always runs last; reserved — packages should not use this)

Within a stage, providers are sorted by `depends` using a topological sort.

### 2. Manifest Format

Generated by `pnpm rudder providers:discover`, written to `bootstrap/cache/providers.json` (gitignored — like Laravel's `bootstrap/cache/packages.php`):

```json
{
  "version": 1,
  "generated": "2026-04-07T12:34:56.000Z",
  "providers": [
    {
      "package": "@rudderjs/log",
      "provider": "log",
      "config": "log",
      "stage": "foundation",
      "depends": []
    },
    {
      "package": "@rudderjs/cache",
      "provider": "cache",
      "config": "cache",
      "stage": "infrastructure",
      "depends": ["@rudderjs/log"]
    }
  ]
}
```

The manifest is **already sorted** in boot order — runtime just iterates.

### 4. Runtime Loader

```ts
// packages/core/src/default-providers.ts
import { resolveOptionalPeer } from './support.js'
import { BUILTIN_REGISTRY } from './provider-registry.js'

export interface DefaultProvidersOptions {
  /** Skip these packages even if installed */
  skip?: string[]
  /** Add these providers after the auto-discovered ones */
  add?: ProviderClass[]
}

export async function defaultProviders(
  configs: Record<string, unknown>,
  options: DefaultProvidersOptions = {},
): Promise<ProviderClass[]> {
  // 1. Try to load the manifest first
  let entries: ProviderEntry[]
  try {
    const manifestPath = path.join(process.cwd(), 'bootstrap/cache/providers.json')
    const manifest = JSON.parse(await fs.readFile(manifestPath, 'utf-8'))
    entries = manifest.providers
  } catch {
    // 2. Fall back to built-in registry, sort it ourselves
    entries = sortByStageAndDepends(BUILTIN_REGISTRY)
  }

  // 3. Filter installed + skipped
  const skip = new Set(options.skip ?? [])
  const installed = await filterInstalled(entries.filter(e => !skip.has(e.package)))

  // 4. Resolve and instantiate each
  const providers: ProviderClass[] = []
  for (const entry of installed) {
    const mod = await import(entry.package)
    const factory = mod[entry.provider]
    if (typeof factory !== 'function') {
      throw new Error(`[RudderJS] ${entry.package} declared provider "${entry.provider}" but it's not exported`)
    }
    const config = entry.config ? configs[entry.config] : undefined
    providers.push(factory(config))
  }

  // 5. Append user-added providers
  if (options.add) providers.push(...options.add)

  return providers
}

async function filterInstalled(entries: ProviderEntry[]): Promise<ProviderEntry[]> {
  const result: ProviderEntry[] = []
  for (const entry of entries) {
    try {
      await resolveOptionalPeer(entry.package)
      result.push(entry)
    } catch {
      if (!entry.optional) {
        // Required package not installed — that's OK, just skip silently
      }
    }
  }
  return result
}
```

### 5. Discover Command

```ts
// packages/cli/src/commands/providers-discover.ts
import { Command } from '@rudderjs/rudder'
import fs from 'node:fs/promises'
import path from 'node:path'

export class ProvidersDiscoverCommand extends Command {
  static signature = 'providers:discover'
  static description = 'Scan node_modules for RudderJS packages and generate the provider manifest'

  async handle(): Promise<void> {
    const cwd = process.cwd()
    const nodeModules = path.join(cwd, 'node_modules')
    const entries: ProviderEntry[] = []

    // Scan @rudderjs scope first, then any other scopes the user has
    const scopes = await fs.readdir(nodeModules)
    for (const scope of scopes) {
      if (!scope.startsWith('@')) continue
      const scopeDir = path.join(nodeModules, scope)
      const packages = await fs.readdir(scopeDir)
      for (const pkg of packages) {
        const pkgJsonPath = path.join(scopeDir, pkg, 'package.json')
        try {
          const pkgJson = JSON.parse(await fs.readFile(pkgJsonPath, 'utf-8'))
          if (pkgJson.rudderjs) {
            entries.push({
              package: pkgJson.name,
              ...pkgJson.rudderjs,
            })
          }
        } catch { /* skip unreadable */ }
      }
    }

    const sorted = sortByStageAndDepends(entries)
    const manifest = {
      version: 1,
      generated: new Date().toISOString(),
      providers: sorted,
    }

    const cacheDir = path.join(cwd, 'bootstrap/cache')
    await fs.mkdir(cacheDir, { recursive: true })
    await fs.writeFile(
      path.join(cacheDir, 'providers.json'),
      JSON.stringify(manifest, null, 2),
    )

    this.info(`✓ Discovered ${sorted.length} provider(s)`)
    for (const e of sorted) {
      this.line(`  ${e.package} (${e.stage})`)
    }
  }
}
```

### 6. Topological Sort

```ts
// packages/core/src/provider-sort.ts
const STAGE_ORDER = ['foundation', 'infrastructure', 'feature', 'monitoring', 'application']

export function sortByStageAndDepends(entries: ProviderEntry[]): ProviderEntry[] {
  // 1. Group by stage
  const byStage: Record<string, ProviderEntry[]> = {}
  for (const e of entries) {
    if (!byStage[e.stage]) byStage[e.stage] = []
    byStage[e.stage]!.push(e)
  }

  // 2. Within each stage, topo-sort by depends
  const sorted: ProviderEntry[] = []
  for (const stage of STAGE_ORDER) {
    const stageEntries = byStage[stage] ?? []
    sorted.push(...topoSort(stageEntries))
  }
  return sorted
}

function topoSort(entries: ProviderEntry[]): ProviderEntry[] {
  const map = new Map(entries.map(e => [e.package, e]))
  const visited = new Set<string>()
  const result: ProviderEntry[] = []

  function visit(entry: ProviderEntry, path: Set<string>): void {
    if (visited.has(entry.package)) return
    if (path.has(entry.package)) {
      throw new Error(`[RudderJS] Circular dependency: ${[...path, entry.package].join(' → ')}`)
    }
    path.add(entry.package)
    for (const dep of entry.depends ?? []) {
      const depEntry = map.get(dep)
      if (depEntry) visit(depEntry, path)
    }
    path.delete(entry.package)
    visited.add(entry.package)
    result.push(entry)
  }

  for (const entry of entries) visit(entry, new Set())
  return result
}
```

## Open Questions

### 1. Should `providers:discover` run automatically?

**Options:**
- **A. Manual** — User runs `pnpm rudder providers:discover` after install. Most explicit.
- **B. postinstall hook** — Add to `package.json` scripts. Auto-runs but might fail in CI environments.
- **C. Vite plugin** — Run during `vite build` and `vite dev`. Always fresh. Slightly slower dev startup.
- **D. All three** — Manual command for explicit, postinstall for one-time setup, Vite plugin for dev mode.

**Recommendation:** Start with **A** (manual). Add **C** (Vite plugin) in a follow-up if it becomes a pain point.

### 2. What about packages with no factory function (e.g. `@rudderjs/socialite` is facade-only)?

**Options:**
- **A. Skip them** — don't add `rudderjs` field. Users import facades directly.
- **B. Add a no-op provider** — just for consistency.

**Recommendation:** **A**. Facade-only packages don't need a provider.

### 3. How to handle multi-driver packages (orm-prisma vs orm-drizzle)?

Both packages export `database` but only one should be active. Options:
- **A. Mark both `optional`** — first installed wins.
- **B. Conflict detection** — error if both manifested.
- **C. Use config to choose** — `configs.database.driver === 'prisma'` → use `@rudderjs/orm-prisma`.

**Recommendation:** **B** + **C**. Detect conflict, error with helpful message. Reading driver from config requires more wiring — defer.

### 4. Versioning the manifest format

Field `version: 1` in the manifest. If the loader sees an unsupported version, it errors with "regenerate manifest". Bumps when adding new fields like `provider2`, `extras`, etc.

### 5. Where exactly does `bootstrap/cache/providers.json` live?

Laravel uses `bootstrap/cache/`. We can match. Add `bootstrap/cache/.gitignore` with `*` so the entire directory is gitignored except itself.

### 6. What happens if the manifest is stale?

If a package was uninstalled but is still in the manifest, the dynamic `import()` will throw. We should:
- Catch the error
- Log a warning: "Package X listed in manifest but not installed. Run `pnpm rudder providers:discover`"
- Skip and continue

### 7. Integration with `Application.configure()`

Should `Application.configure()` itself call `defaultProviders()` automatically? E.g.:

```ts
Application.configure({
  server: hono(configs.server),
  config: configs,
  // No providers field — auto-discovered
})
```

**Recommendation:** No. Keep `providers` explicit so users see what's being registered. The helper is enough.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Manifest gets out of sync after install | Add discover command to common ops docs; consider postinstall hook |
| Circular dependencies between packages | Topo sort throws clearly; document the error |
| Breaking change for existing apps | 100% backward compatible — old manual lists still work |
| Hidden registration order surprises users | Document stage system clearly; log loaded providers in dev mode |
| `resolveOptionalPeer` is slow if scanning many packages | Manifest caches the result so this only happens at discover time |
| Multiple ORM adapters installed | Detect and error with clear message |
