**Status:** ✓ shipped 2026-05-11.

**Prerequisites:** B7 (vector ORM) + B8 (hosted vector stores + `fileSearch` agent tool, OpenAI + local fallback).

## Why this changed from the B8 locked decision

B8's original locked decision punted Gemini hosted RAG to a separate design pass on the assumption that Gemini's surface would be `cachedContent`-shaped — different enough from OpenAI's vector stores that a unified façade would leak.

Google has since shipped `fileSearchStores` on the Gemini Developer API (in the `@google/genai` SDK). It's a direct OpenAI-equivalent: create a store, upload documents with searchable metadata, query via a native `fileSearch` tool block in `generateContent`. The locked-decision concern is resolved — same façade works.

## API parity

The user-facing API is unchanged. The same code works against either provider:

```ts
import { VectorStores, fileSearch } from '@rudderjs/ai'

// OpenAI
const openaiStore = await VectorStores.create('KB', { provider: 'openai' })

// Gemini — same call, different provider
const geminiStore = await VectorStores.create('KB', { provider: 'google' })

await geminiStore.add({ filePath: './report.pdf', attributes: { author: 'Alice', year: 2026 } })

class SupportAgent extends Agent {
  model() { return 'google/gemini-2.5-flash' }
  tools() {
    return [
      fileSearch({
        stores:     [geminiStore.id],     // `fileSearchStores/foo-bar`
        where:      { author: 'Alice', year: { gt: 2020 } },
        maxResults: 10,
      }),
    ]
  }
}
```

## Translation details

**Store id shape.** Gemini resources are full paths (`fileSearchStores/foo-bar`); OpenAI is opaque (`vs_abc123`). Both round-trip through `VectorStoreInfo.id` — apps pass whatever the framework returned back to `get` / `delete` / `addFile`.

**Filter translation.** OpenAI takes typed `{ type: 'and', filters: [{ type: 'eq', key, value }] }`. Gemini takes a string expression: `(author = "Alice") AND (year > 2020)`. The user-facing API is the typed shape; the Google adapter translates internally via `filterToGeminiString`. String values quote-escape via `\"` and `\\`; numbers and booleans render bare.

**Attributes ↔ customMetadata.** Gemini's per-document metadata is an array of `{ key, stringValue|numericValue|stringListValue }`. The framework's flat `Record<string, string | number | boolean>` maps as: strings → `stringValue`, numbers → `numericValue`, booleans → `stringValue: 'true' | 'false'` (Gemini has no boolean variant; string is lossless and filter-matchable). The inverse drops `stringListValue` — no flat representation; apps that need lists can read the raw Document via the SDK directly.

**Ingestion polling.** Gemini uploads return long-running operations. `addFile` polls `client.operations.get({ operation })` at the configured interval (default 1 s) until `done`, then calls `documents.get` to fetch the real `state` / `sizeBytes` / `createTime`. Failed operations surface as `{ status: 'failed', lastError }` without throwing — matches the OpenAI path.

**Unsupported surface.** Gemini has no equivalent of OpenAI's store-level `metadata` or `expiresAfter` policy. Passing either throws fail-loud at adapter time so apps don't silently lose data. Per-document attributes still work normally.

## Provider hint emission

The same `providerHint.type === 'file-search'` that drives OpenAI's native block emits Gemini's native shape in `toGeminiTools`:

```ts
// OpenAI emits:
{ type: 'file_search', vector_store_ids: [...], filters?: { typed object }, max_num_results?: 10 }

// Gemini emits:
{ fileSearch: { fileSearchStoreNames: [...], metadataFilter?: 'string syntax', topK?: 10 } }
```

The `WebSearch` retrofit (B8 Phase 2.x) already established this pattern with `google_search` — B8.5 extends it to `fileSearch`.

## Vertex AI

Gemini's `fileSearchStores` is **Developer API only**. The underlying SDK methods throw `'This method is only supported by the Gemini Developer API'` for Vertex clients. Apps on Vertex need to stand up RAG via Vertex AI Search separately — a future plan if there's demand.

## Tests

11 new test suites (~50 cases) in `packages/ai/src/google-vector-stores.test.ts`:

- `filterToGeminiString` — eq/ne/gt/gte/lt/lte, AND/OR/nested, string-quote escaping, empty-sub-filter throws.
- `attributesToCustomMetadata` + `customMetadataToAttributes` — string/number/boolean coercion, `stringListValue` drop, round-trip.
- `fromGeminiFileSearchStore` — displayName/override/fallback, sizeBytes parsing, ISO timestamps.
- `fromGeminiDocument` — STATE_* enum mapping (ACTIVE/FAILED/PENDING/UNSPECIFIED), customMetadata round-trip.
- `GoogleVectorStoreAdapter` — create (unsupported metadata/expiresAfter throws), list/get/delete, addFile (importFile vs upload paths), LRO polling (sync / async / wait:false / timeout / error), removeFile path normalization, listFiles status mapping.
- `toGeminiTools` — file-search hint emits native `fileSearch` block, forwards `topK` + `metadataFilter`, omits absent fields, falls through for non-hinted tools.

## Out of scope

- **Vertex AI hosted RAG.** Separate API (`vertexAiSearch` / `vertexRagStore`), different auth, different shape. Future plan if there's demand.
- **Anthropic hosted vector stores.** Anthropic has Files but no first-class hosted vector store yet. Retrofit via the same `providerHint` pattern when they ship one.
- **Mixed-provider stores in a single `fileSearch` call.** The framework enforces "one provider per agent" at the registry layer. Apps querying both an OpenAI store AND a Gemini store would need two `fileSearch` tools.
- **`stringListValue` round-trip.** No flat-attribute representation; dropping on read is intentional. Apps that need list-valued metadata read the raw Document via the SDK.
