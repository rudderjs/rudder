# @rudderjs/orm

Eloquent-style ORM — Model base class, casts, attributes, scopes, observers, factories, and API resources. **Built on `@rudderjs/database`** — the native engine (compiler, dialects, drivers, `NativeQueryBuilder`, `NativeAdapter`, schema builder/migrator) physically lives there since the Phase-2 relocation (#889/#891/#892); `src/native/` here keeps only the `index.ts` shim (`export * from '@rudderjs/database/native'`), `provider.ts` (`NativeDatabaseProvider` — wires `ModelRegistry`/`ConnectionManager`/db-bridge, so it stays orm-side), and the Model-coupled engine test suites. `@rudderjs/database` must NEVER depend on `@rudderjs/orm` (devDep included — turbo graph cycle); engine tests that need `Model` live here, pure engine tests live there. See `packages/database/CLAUDE.md`.

## Key Files

- `src/index.ts` — `Model` base class (query, CRUD, scopes, observers, serialization), decorators, `ModelRegistry`, `ModelNotFoundError`
- `src/cast.ts` — Built-in casts (string, integer, float, boolean, date, json, encrypted, **`decimal:N`**, **enum**, **`hashed`**) + custom `CastUsing` + `vector({ dimensions })` factory (#B7 Phase 1) — returns a `CastUsing` class that serializes `number[]` ⇄ pgvector text format with write-time dimension/finiteness validation. **Cast breadth** (Laravel parity): `decimal:N` is a `` `decimal:${number}` `` template-literal cast — read+write both normalize to a fixed-precision *string* (`'9.50'`, no float drift). **enum** = a TS `enum`/const object placed directly in `static casts` (`{ status: StatusEnum }`); detected at runtime by NOT being a `CastUsing` ctor (no `get`/`set` on prototype), validates membership, throws on unknown, and strips TS numeric-enum reverse-mapping keys (`/^\d+$/`) so only forward values are valid. **`hashed`** hashes one-way on write via `@rudderjs/hash`, read **synchronously** through its globalThis-shared registry (`__rudderjs_hash_registry__.driver.makeSync`) — NOT a static/`await import` of `@rudderjs/hash` (node-only; would break the client bundle). No-op when already hashed (driver `isHashed` or a bcrypt/argon2 regex fallback); needs a sync-capable driver (bcrypt — `makeSync` added in `@rudderjs/hash`; argon2 throws). **Why sync:** `castSet` runs in `_applyMutators` which is NOT awaited (the payload flows straight into `compileInsert` bindings), so a cast `set` returning a Promise would store `[object Promise]` — every cast must be synchronous. (The legacy `encrypted` cast's `require('@rudderjs/crypt')` is dead in this ESM package — `require` is undefined at runtime — don't copy that pattern; use the globalThis-registry bridge.)
- `src/vector-errors.ts` — `VectorDimensionMismatchError`, `VectorStorageUnsupportedError`, `MissingEmbedderError` (all carry stable `code` fields for app `instanceof` + `.code` dispatch)
- `src/attribute.ts` — `Attribute.make({ get, set })` for computed accessors/mutators
- `src/collection.ts` — `ModelCollection` typed array wrapper with ORM operations
- `src/factory.ts` — `ModelFactory` for testing with states and sequences
- `src/resource.ts` — `JsonResource` / `ResourceCollection` for API response transformation: conditional helpers (`when`/`whenNotNull`/`whenLoaded`/`mergeWhen`/`whenHas`/`whenCounted`/`whenAggregated`), paginator-aware `collection()`, `additional()`, `toResponse()` envelopes. Client-reachable — no `node:` imports, no Model imports (one-way dependency).
- `src/connection-manager.ts` — `ConnectionManager`: lazy named-connection registry (multi-connection support)
- `src/deferred-connection-qb.ts` — record-and-replay QB for lazily-opened named connections (`static connection` / `Model.on()` first query)
- `src/sticky.ts` — re-export shim of `@rudderjs/database/sticky` (the scope moved with the engine; same globalThis key `__rudderjs_orm_sticky__`, so both paths share one scope). Node-only — NEVER re-export from the main entry
- `src/seeder.ts` — `Seeder` base class for `db:seed` runner
- `src/commands/make-factory.ts` — `makeFactorySpec` MakeSpec for `pnpm rudder make:factory <Name>` (writes `app/Factories/<Name>Factory.ts`)
- `src/commands/make-seeder.ts` — `makeSeederSpec` MakeSpec for `pnpm rudder make:seeder <Name>` (writes `database/seeders/<Name>Seeder.ts`)
- `src/commands/make-resource.ts` — `makeResourceSpec` MakeSpec for `pnpm rudder make:resource <Name>` (writes `app/Resources/<Name>Resource.ts`; model import inferred from the stem)

## Architecture Rules

- **Adapter pattern**: `ModelRegistry.set(adapter)` plugs in the actual DB driver (Prisma adapter lives in playground)
- **Per-model named connections (multi-connection PR2)**: `static connection = 'reporting'` rebinds every query of the model; `Model.on('reporting')` is the one-off form. **`Model.on()` is overloaded by arity** — two-arg = lifecycle listener (`User.on('creating', fn)`, unchanged), one-arg = connection-scoped query; don't collapse the overloads. Routing lives in `Model._adapterQb()` (used by both `query()` and `_q()`): tx-scoped adapter first (`getScopedAdapter`) → opened adapter (`ConnectionManager.peek`) → **deferred record-and-replay QB** (`deferred-connection-qb.ts`) that queues chainables and opens the connection at the first Promise-returning terminal (terminal list = the adapter `QueryBuilder` contract's Promise members — keep it in sync if the contract gains terminals). After materialization the proxy forwards directly (so `chunk`/`lazy`'s limit/offset mutation cycles hit the real builder); only the FIRST query per connection pays the recorder. Known limits (self-diagnosing throws, not silent): a recorded method missing on the materialized adapter QB throws naming the method; `union()` members must come from an already-open connection. The default-connection path pays only one `undefined` check — don't add named-connection work to it.
- **Read/write split + sticky (multi-connection PR3, native engine)**: a connection config may set `read: { url: string | string[] }` (replicas, round-robin per query), `write: { url }` (defaults to top-level `url`), `sticky: true`. Routing: un-locked SELECT terminals + `selectRaw` → read pool via the adapter's `readPick` closure (passed into `NativeQueryBuilder` as the 5th ctor param; `_readExecutor()` bypasses it when `_lock` is set); writes, DDL, locked selects, and ALL transaction statements → write executor (`_reselect` deliberately stays write-side — read-your-write after a no-RETURNING insert). **Sticky** = ALS scope in `src/sticky.ts` (globalThis `__rudderjs_orm_sticky__`; `runWithDatabaseContext`/`markWrote`/`stickyWrote`); the WRITE executor of a split connection is wrapped in `markWritesExecutor` (marks BEFORE executing; marking locked selects/sticky reads is deliberate over-approximation), and `readPick` returns the write executor on a sticky hit. The provider auto-installs `databaseContextMiddleware()` on web+api when a sticky split connection exists; outside a scope sticky is a no-op (jobs/commands read replicas — Laravel resets per request anyway). Query events: `connection` = config connection name (fallback driver name), `target: 'read'|'write'` ONLY on split connections (contracts `QueryEvent.target`). HMR cache entry holds `readDrivers` (signature includes the replica list; supersede/disconnect close them). Tests prove routing by data divergence across two seeded sqlite files — reuse that trick. **Drizzle adapter (Task 4b): same config surface + same routing rules, now real** — implemented at the QB level since Drizzle has no Executor seam to wrap: a `DrizzleSplitInfo` (read picker, sticky marker, event tag, connection name) is threaded into every builder; `_readDb()` picks per read terminal (write db on lock/sticky-hit), `_writeDb()` marks sticky BEFORE exec at every write-terminal head, the raw `selectRaw`/`affectingStatement` seams route too, and a tx-scoped adapter inherits the `'write'` tag + sticky marking. Replica clients open via the factored `openDrizzleClient()`; the Drizzle HMR cache entry holds `readDbs`. Drizzle events report `connection` = `connectionName ?? dialect`. Raw-seam fakes + two-file sqlite routing tests in `orm-drizzle/read-write-split.test.ts` (drizzle's sqlite wrappers have no `db.execute`, so raw seams use execute-capable fake clients). **Prisma**: `read:`/`write:` config still throws at boot → `@prisma/extension-read-replicas`.
- **Named connections (multi-connection PR1)**: `ConnectionManager` holds **lazy factories** per connection name, registered by each adapter provider's `boot()` — registering does NO I/O and no driver import (`config/database.ts`'s `connections` is a MENU; an unused entry must never open or import its optional-peer driver). `ensure(name)` opens single-flighted + memoized; `peek(name)` is the sync no-open read. The DEFAULT connection keeps its eager `ModelRegistry.set` path — the native provider resolves it **through the same ConnectionManager entry** so `DB.connection(default)` and Models share one adapter/driver. The transaction ALS stores a **`ReadonlyMap<connectionName, OrmAdapter>`** (not a single adapter): `transaction(fn, { connection: 'x' })` scopes only `'x'`, so named-connection transactions never capture default queries (per-connection isolation, Laravel parity); nested named transactions resolve the outer scoped adapter first → SAVEPOINT. `@rudderjs/database`'s `DB.connection(name)` resolves via `registerConnectionResolver` pushed from `db-bridge.ts` (scoped-adapter check first, so `DB.connection(name).*` joins an open named transaction). The native dev-HMR driver cache (`__rudderjs_native_client__`) is now a **per-connection Map** keyed by `connectionName ?? signature` — `disconnect()` evicts by driver identity; don't revert to the single-entry shape (a second named connection would evict/close the first). `connection-manager.ts` is on the client-reachable graph — no `node:` imports there.
- **Event lifecycle**: `Model.create/update/delete()` trigger observer events; `query().create()` bypasses them. Events: `retrieved`, `creating`/`created`, `updating`/`updated`, `saving`/`saved` (fire on both create+update), `deleting`/`deleted`, `restoring`/`restored`. `Model.withoutEvents(fn)` mutes per-class. **`increment`/`decrement` deliberately do NOT fire observer events** — they're a pure data-plane operation; the observer payload would have to be either the delta or the resolved value, both of which conflict with atomic SQL semantics.
- **Serialization order**: casts (get) → attributes (getter) → appends → visible/hidden filtering → `toJSON()`
- **API resources** (`src/resource.ts`): `Resource.collection()` accepts paginator results directly and derives the envelope `meta` — **duck-typed** detection (`OffsetPaginated` needs numeric `total`/`perPage`/`currentPage`/`lastPage`; `CursorPaginated` needs `nextCursor`/`prevCursor` keys + boolean `hasMore`), never `instanceof` (dev-HMR re-import fragility). Offset meta emits `page` ← `currentPage`. Explicit `meta` arg merges over derived. `additional(extra)` (both classes) merges extra TOP-LEVEL envelope keys; the envelope's own `data`/`meta` win on conflict. `JsonResource.toResponse()` = `{ data, ...additional }`, async-safe (`toJSON()` throws on async `toArray()` — by design). `whenCounted`/`whenAggregated` read the stamped aggregate aliases via **`aggregateAlias()` from `aggregate.ts`** — never re-derive the camelCase rules. `whenPivotLoaded` deliberately absent (gated on pivot-column reads). Model wiring: `static resourceClass` + `instance.toResource(cls?)` + `ModelCollection.toResourceCollection(cls?)` — explicit class wins, unbound throws a pointer error, empty collection needs no class; mirrors `factoryClass` (no name-convention auto-discovery). Generic constraints are `T extends object` (NOT `Record<string, unknown>` — class instances lack index signatures); `resource.ts` must stay free of Model imports (one-way dependency, `collection.ts` runtime-imports resource, never the reverse).
- **Mass assignment**: `static fillable` (allowlist) and `static guarded` (denylist; `['*']` locks all) are enforced on `Model.create()`, `Model.update()`, and `instance.fill()` — keys outside the policy are silently dropped. Both default to `[]` (no enforcement, back-compat). `fillable` wins when both are set. Bypass paths: `instance.forceFill(data)` and direct property assignment + `instance.save()` (data set property-by-property is not mass-assignment, so `save()` skips the filter via internal `_doCreate`/`_doUpdate`). **`Model.upsert()` and `insertMany` also bypass the filter** (bulk ops) — write-side casts/mutators still apply on upsert.
- **`chunk(size, cb)` / `lazy(size?)`** — memory-bounded iteration, implemented in the `_hydratingQb` proxy (Model-layer only; NO contracts/adapter change — they page via the existing `limit`/`offset`/`get` primitives, mutating `target`'s limit/offset each pass and reusing the `get` hydration path). `chunk` re-queries per page (offset paging), returns `true`/`false` (callback `false` bails); `lazy` is an async generator (default page 1000). Both also exposed as Model statics. Offset-paging relies on a consistent sort — recommend an `orderBy` (mirrors Laravel `chunk`, which also doesn't auto-order). Typed on `HydratingQueryBuilder` (NOT the `QueryBuilder` contract). Tests in `chunk-lazy.test.js` (own file in the orm test-list).
- **`Model.upsert(rows, uniqueBy, update?)`** — bulk insert-or-update (Laravel parity). Model layer normalizes `uniqueBy`→array, computes the default `update` set (every inserted column minus `uniqueBy`), applies write-time casts/mutators per row, then dispatches to the adapter's optional `QueryBuilder.upsert(rows, uniqueBy[], update[])`. Native + Drizzle emit ONE statement (`ON CONFLICT … DO UPDATE`/`DO NOTHING` on sqlite/pg via the new `Dialect.upsertClause()` seam + `compileInsert({upsert})`; `ON DUPLICATE KEY UPDATE` on MySQL — `uniqueBy` ignored there since MySQL keys off any unique index). Prisma has no portable bulk ON CONFLICT → per-row `delegate.upsert` batched in one `$transaction`. **No observer events fire** (pure data-plane, like `increment`/`insertMany`). Empty `update` = insert-or-ignore. `upsert?` is OPTIONAL on the `OrmAdapter`/`QueryBuilder` contract — the Model layer throws an adapter-named error if absent. MySQL's returned count is rows-*touched* (1/insert, 2/update), not distinct.
- **Raw expressions (`selectRaw`/`whereRaw`/`orWhereRaw`/`orderByRaw` + `raw(...)`/`DB.raw(...)`)** — raw-SQL escape hatch on the `QueryBuilder` contract (required methods, not `HydratingQueryBuilder`-only). The `Expression` wrapper + `raw()` live in **`@rudderjs/contracts`** (re-exported by `@rudderjs/database` so `DB.raw()` keeps its path) — NOT in `@rudderjs/database`, because the QB raw methods are Model/client-reachable and `@rudderjs/database` is node-only. Raw fragments use `?` placeholders + a positional `bindings` array; the native compiler's `compileRaw` rebinds each `?` to the dialect placeholder (`$n` on pg) through the **one shared positional `Bindings`** — so select-list raw bindings land before WHERE's, which land before ORDER BY's (SQL text order). `?` count must equal `bindings.length` or it throws. `selectRaw` REPLACES the default `*` projection (Laravel semantics). A `raw()` value in a `where(col, op, raw(...))` splices verbatim (no bind); `orderBy(raw(...))` is a raw order item. **Native `_orders` is `OrderItem[]` (structured `OrderClause` | `{kind:'raw'}`) and Drizzle's `_orders` is likewise a union — one list keeps interleaved structured+raw order.** Adapter divergence: **native** = full; **Drizzle** = whereRaw/orWhereRaw/orderByRaw via its `sql` template (the module-level `rawToSql` splits on `?` and parameterizes), `selectRaw` THROWS (typed select can't map a raw projection back to a model); **Prisma** = all four THROW (structured client) pointing at `DB.select(sql, bindings)`. Tests in `native/raw-expr.test.ts` (own line in orm test-list) + `orm-drizzle/raw-expr.test.ts` + `orm-prisma/raw-expr.test.ts`. **Adding a required QB method breaks ~15 hand-rolled QB test stubs (orm + ai) — they each need the new methods.**
- **`whereX` query sugar** (`whereIn`/`whereNotIn`/`whereNull`/`whereNotNull`/`whereBetween`/`whereNotBetween` + `orWhere*` forms, `when`/`unless`, `latest`/`oldest`, and the scalar terminals `pluck`/`value`/`sum`/`max`/`min`/`avg`/`exists`/`doesntExist`) — implemented ENTIRELY in the `_hydratingQb` proxy (Model-layer; like `chunk`/`lazy`), composing the existing `where`/`orWhere`/`whereGroup`/`orderBy`/`get`/`first`/`_aggregate` primitives. **NO contracts/adapter change → all 3 adapters get them free, and no QB-stub churn.** `whereBetween`→grouped `>=a AND <=b`; `whereNotBetween`→grouped `<a OR >b`; `whereNull`→`where(col,'=',null)` (compiler emits IS NULL). `when(v,cb,else?)`/`unless` pass `(proxy, value)` to the callback. `pluck`/`value` hydrate via `wrapMany`/`wrapMaybe` so casts apply; `sum`/`max`/`min`/`avg`/`exists` route through the QB `_aggregate(fn, col?)` terminal (`exists` returns boolean). Typed on `HydratingQueryBuilder` (NOT the `QueryBuilder` contract); each also a `Model` static entry point. Tests in `where-sugar.test.js` (own orm test-list line). The former deferrals are all closed now: structured `select()`/`distinct()` (see their bullets), and `whereNot` group-negation + date helpers (see the next bullet).
- **Date helpers (`whereDate`/`whereTime`/`whereDay`/`whereMonth`/`whereYear` + `orWhere*` forms) + `whereNot(cb)`/`orWhereNot(cb)`** — NOT pure Model-layer sugar (need real SQL): **NATIVE + DRIZZLE real; Prisma throws a clear Model-layer guard error** (`[RudderJS ORM] <method>() is not supported on this adapter — use whereRaw(...) or DB.select(...)`) from the `_hydratingQb` proxy when the adapter QB lacks the method (forward-or-throw). Date helpers: two-arg = equality, three-arg carries the operator (Laravel semantics); a new `'date'` `ConditionNode` kind compiles through the per-dialect **`Dialect.dateExtract(part, quotedColumn)`** seam — sqlite `strftime('%Y-%m-%d'|'%H:%M:%S', col)` with `CAST(... AS INTEGER)` for day/month/year (strftime returns zero-padded TEXT and SQLite never equates TEXT with INTEGER), pg `col::date`/`col::time`/`EXTRACT(DAY|MONTH|YEAR FROM col)::int`, mysql `DATE()/TIME()/DAY()/MONTH()/YEAR()`. Values bind through the one shared positional `Bindings`. Value normalization in the native QB (`normalizeDatePartValue`): `Date` → **UTC** components (`'YYYY-MM-DD'`/`'HH:MM:SS'`/ints); numeric strings on day/month/year → `Number` (so `'05'` matches the INTEGER extraction). `whereNot`/`orWhereNot` reuse the `whereGroup` sub-builder machinery — the `group` `ConditionNode` gained a `negated?: boolean` compiled as `NOT (…)`; empty callback = no-op; the proxy wraps the sub-builder hydrating so named sugar (`whereIn` etc.) composes inside the callback. Methods on `HydratingQueryBuilder` + `Model` statics (AND forms only) — NOT the `QueryBuilder` contract, so **zero QB-stub churn**. Tests: `native/date-helpers.test.js` (compiler units pinning SQL per dialect + sqlite E2E + gated live-pg) + `native/where-not.test.js`. **Drizzle impl** (`orm-drizzle`): same surface, expressions built via the `sql` template per dialect (mirrors `dateExtract`; local copy of `normalizeDatePartValue` — the native one is node-only); `whereNot` wraps the sub-builder's `buildConditions()` in Drizzle's `not()`; pushed onto `_extraExprs`/`_orExtraExprs` like whereRaw/whereColumn. Tests: `orm-drizzle/date-helpers-not.test.ts` (sqlite E2E + pg/mysql SQL shapes pinned via the `pg-proxy`/`mysql-proxy` drivers — no server needed).
- **JSON-path predicates (arrow `where()` + `whereJsonContains`/`whereJsonDoesntContain`/`whereJsonLength` + `orWhere*` forms)** — **NATIVE + DRIZZLE real; Prisma throws the forward-or-throw guard error**. Arrow detection lives in the native QB's `_pushClause` (`column.includes('->')`), so plain `where('meta->prefs->lang', 'en')`/`orWhere` AND everything composed from them (group callbacks, `whereNot`, `whereIn`/`whereNull`/`whereBetween` sugar) get JSON paths for free; the named methods parse via the same `parseJsonPath`. Three new `ConditionNode` kinds (`json`/`jsonContains`/`jsonLength`, same shape as `'date'`) compile through per-dialect seams **`Dialect.jsonExtract(quotedCol, segments, valueKind)` / `jsonContains(quotedCol, segments, value, bind)` / `jsonLength(quotedCol, segments)`** (+ `jsonBoolean(v)` for value normalization): sqlite `json_extract(col, '$."a"."b"')` (typed results — booleans normalize to `1`/`0`), pg arrow chains `col->'a'->>'b'` with **`::numeric`/`::boolean` casts when the comparison value is a number/boolean** (text `'10' < '9'` footgun) + `(col->'tags')::jsonb @> $n::jsonb` + `jsonb_array_length`, mysql `JSON_UNQUOTE(JSON_EXTRACT(...))` (**booleans skip UNQUOTE and splice the SQL literal `true`/`false`** — Laravel's grammar shape; a BOUND 'true' string coerces to a JSON *string* in comparison context and matches NOTHING, verified live on MySQL 8) + `JSON_CONTAINS` + `JSON_LENGTH`. **Injection boundary**: path segments are user strings spliced into SQL — `parseJsonPath` REJECTS quotes/backslashes/backticks/control chars (spaces+dots allowed via `$."a b"` quoting); all-digit segments become array indexes (`meta->items->0` → `$[0]` / pg `->0`). sqlite `whereJsonContains` is emulated via `json_each` EXISTS per element — scalars only (object values throw); array values = every element contained (matches pg `@>`/mysql). The comparison tail (`Expression`/IN/IS NULL) is shared with plain clauses via the extracted `compileComparison(lhs, op, value, b)` — so `whereNull('meta->x')`/`whereIn('meta->x', [...])` compose. The proxy guard set was renamed `DATE_PART_METHODS` → **`FORWARDED_QB_METHODS`** (date + JSON methods). Methods on `HydratingQueryBuilder` + `Model` statics (AND forms only) — NOT the `QueryBuilder` contract, **zero QB-stub churn**. **Null semantics (Laravel parity)**: `whereNull('meta->x')`/`where('meta->x', null)` matches a MISSING key AND an explicit json `null` on every dialect — sqlite/pg extractions surface both as SQL NULL, but mysql's `JSON_EXTRACT` returns a JSON null LITERAL for an explicit null (and UNQUOTE'd it's the STRING `'null'`), so null equality routes through the **`Dialect.jsonNullComparison(col, segments, negated)`** seam — mysql emits Laravel's `(extract IS NULL OR JSON_TYPE(extract) = 'NULL')` (negation ANDs the inverses; `JSON_TYPE(NULL)` is NULL so the `!=` arm alone is three-valued); sqlite/pg keep the plain `IS [NOT] NULL` byte-identical. Same shape inlined in the Drizzle adapter's `_jsonClauseExpr` mysql branch. (Closed since: `->` inside `whereHas` constrain callbacks works on native too (#878); `update()` to a JSON path — see the JSON-path updates bullet.) **Driver fix that rode this PR**: porsager's default json type `JSON.stringify`s every param the server describes as json/jsonb, so the pg dialect's pre-stringified `@>` binding was DOUBLE-encoded and containment matched nothing — `PostgresDriver` now overrides the `json` type (strings verbatim, non-strings stringify; same pattern as the #860 `date` override). Tests: `native/json-where.test.js` (parse/injection units + per-dialect SQL pins + sqlite E2E + gated live-pg + gated live-mysql). **Drizzle impl** (`orm-drizzle`): same surface; arrow detection intercepts at `clauseToExprOn` (the ONE clause-render funnel) so groups/whereNot/sugar AND `whereHas` constraint callbacks (rendered against the related table — works on Drizzle, still deferred on native) compose for free; local copies of `parseJsonPath`/`jsonPathLiteral` (node-only boundary, same call as `normalizeDatePartValue`); pg containment binds through `cast(? as text)::jsonb` (a param described jsonb gets re-stringified by postgres-js — the same double-encode the native driver had to fix; node-postgres would fail a bare-string parse; the text cast behaves identically on both); mysql booleans = spliced literal via `sql\`true\``. Tests: `orm-drizzle/json-where.test.ts` (sqlite E2E + pg/mysql proxy SQL pins + gated live-pg/live-mysql).
- **JSON-path updates (arrow keys in `update()`/`updateAll()` payloads)** — **NATIVE + DRIZZLE real; Prisma throws a clear Model-layer error** (the hydrating proxy checks the QB's `supportsJsonPathUpdates` marker — falsy-based so the deferred named-connection recorder passes through). `Model.update(id, { 'meta->prefs->lang': 'en' })` writes ONE path inside the JSON column: native via the **`Dialect.jsonSet(column, writes, bind)`** seam (`compileJsonSetClause` — sqlite `json_set(col, path, json(?))`, mysql `JSON_SET(col, path, CAST(? AS JSON))`, pg nested `jsonb_set((col)::jsonb, ARRAY[…], $n::jsonb)`); Drizzle via `_jsonSetValues()` building per-column `sql` exprs with the same shapes — EXCEPT pg binds `cast(? as text)::jsonb` (postgres-js re-stringifies jsonb-described params, the #874 double-encode trap; the native driver instead fixed its own `json` type). Every value binds as JSON text (`JSON.stringify`) so all types round-trip identically; multiple arrow writes on one column merge into a single assignment; whole-column + arrow write on the same column THROWS (would race last-one-wins); path segments run the same `parseJsonPath` injection gate as reads; plain payloads compile byte-identical (zero-cost gate). Under `fillable`/`guarded` the arrow key itself must be listed (Laravel parity). `UpdatePayload<T>` template-literal type admits `col->path` keys on `HydratingQueryBuilder.update`/`updateAll` + `Model.update`. Tests: `native/json-update.test.ts` + `orm-drizzle/json-update.test.ts` (sqlite round-trip + pg/mysql proxy SQL pins + gated live-pg/live-mysql).
- **`whereColumn` / `orWhereColumn`** (`whereColumn('a','b')` / `whereColumn('a','>','b')`) — compare two columns, both sides identifier-quoted per dialect (the reason it can't ride on `whereRaw`, which is verbatim). NOT pure Model-layer sugar — needs real SQL: native adds a `'column'` `ConditionNode` kind compiled as `quoteId(left) op quoteId(right)`; **Drizzle** renders both as column refs through `sql` (quoted per dialect); **Prisma** throws → `DB.select`/`whereRaw` (no column-vs-column in its query API). Methods live on `HydratingQueryBuilder` (required) + `Model` statics — NOT on the `QueryBuilder` contract, so **zero QB-stub churn**. Tests: `native/where-column.test.js` (own orm test-list line) + drizzle/prisma.
- **Application-generated primary keys (`static keyType`)**: `'int'` (default) = database auto-increment, ORM never sets the PK. `'uuid'` / `'ulid'` = the Model layer stamps a fresh key on `_doCreate` (covers `create`/`save`/`firstOrCreate`/`updateOrCreate`) when the PK is unset — UUID v4 via Web Crypto `randomUUID`, ULID via a 26-char Crockford Base32 generator (`src/keys.ts`, 48-bit ms timestamp prefix → lexicographically sortable). Both use `globalThis.crypto` so `keys.ts` stays client-bundle safe (no `node:` import). Pure Model-layer (in `_ensureGeneratedKey`, called from `_doCreate`) → all 3 adapters free, no contract change. A caller-supplied key is never overwritten. Bulk `insertMany`/`upsert` are NOT covered (pure data-plane, like their observer-skipping). Tests: `key-type.test.js` (own orm test-list line).
- **Joins + structured `select()`** (`join`/`leftJoin`/`rightJoin`/`crossJoin` + `select(...columns)`) — **NATIVE engine only**; **Drizzle + Prisma THROW** with a pointer to the native engine / `DB.select(...)` (same result-shape-mapping reason `selectRaw` throws — a join/projection changes the row shape the typed clients map back to one model). Methods live on `HydratingQueryBuilder` + `Model` statics, **NOT the `QueryBuilder` contract → zero QB-stub churn** (the proxy's `default` case auto-forwards them to the adapter QB and re-wraps `this`→`proxy`). RIGHT JOIN needs SQLite 3.39+. - **`groupBy` / `having`** (`groupBy(...cols)` + `having(col,op,val)`/`orHaving`/`havingRaw(sql,binds)`/`orHavingRaw`) — **NATIVE engine only**; **Drizzle + Prisma THROW** (same `_builderUnsupported` pointer as joins). `NativeQueryState` gains `groupBy?: string[]` + `having?: HavingNode[]` (`HavingNode` = `clause` (binds) | `raw`). Methods on `HydratingQueryBuilder` + `Model` statics (`groupBy`/`having`/`havingRaw`), NOT the contract → zero stub churn. Tests: `native/group-having.test.js` (own orm test-list line). **UPDATE: Drizzle now REAL** — `groupBy()`→`.groupBy(resolveColumn(...))`, `having`/`orHaving`/`havingRaw`/`orHavingRaw` build a Drizzle `SQL` (mirrors `buildConditions`' AND/OR shape) → `.having(...)`. `count()`/`paginate()` of a grouped builder wrap the projection as a `.as('aggregate')` subquery and `COUNT(*)` it (the count path projects just the GROUP BY keys — portable, avoids `SELECT * … GROUP BY` strict-dialect rejection). Aggregate projections still need `selectRaw` (throws on Drizzle) → HAVING on an aggregate uses `havingRaw('COUNT(*) > ?')`, not `having('alias', …)`. Tests: `orm-drizzle/group-having-distinct.test.ts`.
- **`union` / `unionAll`** (`base.union(other)` / `base.unionAll(other)`) — **NATIVE engine only**; **Drizzle + Prisma THROW** (`_builderUnsupported`). `NativeQueryState` gains `unions?: {all,state}[]`. `compileSelect` was refactored to extract **`compileSelectBody`** (projection→HAVING, no ORDER/LIMIT/lock) — the base body + each member body are joined with `UNION [ALL]` over the **one shared positional `Bindings`** (base binds first, then members in order), then the **base query's** ORDER BY / LIMIT / OFFSET / lock apply to the whole result (member ORDER/LIMIT are dropped). `compileCount` wraps the union (`SELECT COUNT(*) FROM (… UNION …) AS aggregate`) so count()/paginate() count combined rows (takes precedence over the GROUP BY wrap). The no-union path is byte-identical to before (regression-safe). **Unwrap mechanism:** `union(other)` receives the `HydratingQueryBuilder` Proxy and reads its underlying native builder via `Symbol.for('rudderjs.orm.qb.target')` — the proxy answers that symbol with its `target`; defined independently (NOT imported) in both `index.ts` and `query-builder.ts` so the node-only native module stays out of the client-reachable import graph. Members must be native builders (throws otherwise). Methods on `HydratingQueryBuilder` only (no Model static — union starts from a base query). Tests: `native/union.test.js` (own orm test-list line). **UPDATE: Drizzle now REAL** — built on Drizzle's native `.union()`/`.unionAll()` set operators (a post-union `orderBy`/`limit`/`offset` attaches to the whole compound; Drizzle auto-unqualifies order columns to plain identifiers as set-operation SQL requires). Members unwrap via the same `Symbol.for('rudderjs.orm.qb.target')` proxy answer and contribute their select body via `_selectBodyQuery()` (projection → HAVING, no ORDER/LIMIT); `count()`/`paginate()` wrap the compound as a `.as('aggregate')` subquery and `COUNT(*)` it (precedence over the GROUP BY/DISTINCT wrap). **This closed the LAST throwing Drizzle QB method** — native + Drizzle query-builder parity is complete (`selectRaw` is the one remaining DB-facade pointer on Drizzle; Prisma intentionally points builder-SQL at `DB.select`). Tests: `orm-drizzle/union.test.ts`.
- **`distinct()`** — `SELECT DISTINCT`; **NATIVE engine only** (Drizzle+Prisma THROW via `_builderUnsupported`). `NativeQueryState.distinct?: boolean`; `compileSelectBody` emits `SELECT DISTINCT …`. `compileCount` wraps the DISTINCT body (`COUNT(*) FROM (SELECT DISTINCT …) AS aggregate`) so `distinct().count()` counts distinct rows (a bare `COUNT(DISTINCT *)` isn't valid SQL); no-distinct path unchanged. `HydratingQueryBuilder` + `Model.distinct()` static. Tests: `native/distinct.test.js` (own orm test-list line). **Closes the `select()`/`distinct()` deferral fully.** **UPDATE: Drizzle now REAL** — `distinct()` flips `_distinct`; the projection root becomes `db.selectDistinct(fields)` (vs `db.select`). `distinct().count()` wraps the distinct body as a subquery and `COUNT(*)`s it (counts distinct rows). Tests: `orm-drizzle/group-having-distinct.test.ts`.
- **CTEs (`withExpression` / `withRecursiveExpression`)** — **NATIVE engine only** (Drizzle/Prisma throw via the `FORWARDED_QB_METHODS` forward-or-throw guard). `withExpression(name, query, opts?)` where `query` = another native query (proxy-unwrapped via the same `Symbol.for('rudderjs.orm.qb.target')` as `union()`) or a raw SQL string + `opts.bindings` (recursive bodies are usually raw — they reference the CTE's own name, which a table-rooted builder can't express); `opts.columns` → explicit column list. `NativeQueryState.ctes` compiles via `compileCtePrefix` to a `WITH [RECURSIVE] name [(cols)] AS (body), …` prefix on `compileSelect` AND every `compileCount` variant — **CTE bindings land FIRST** (text order, before SELECT-list/join/WHERE binds) through the one shared positional `Bindings`; one recursive member marks the whole list `RECURSIVE`. Builder-backed bodies keep their own UNION members but drop ORDER/LIMIT (same rule as `union()`); a builder body with `opts.bindings` throws. Reference the CTE via `join('name', …)` — the FROM stays the model table. Read-side only (get/first/find/count/paginate). Methods on `HydratingQueryBuilder` + `Model` statics, NOT the contract → zero QB-stub churn. Tests: `database/src/native/cte.test.ts` (compiler units + adapter-level sqlite E2E) + `orm/src/native/cte.test.ts` (Model layer + gated live-pg recursive + guard).
- **Window functions (`selectWindow`)** — typed zero-arg ranking set (`rowNumber`/`rank`/`denseRank`/`percentRank`/`cumeDist`) → `FN() OVER (PARTITION BY … ORDER BY …) AS alias`; **NATIVE engine only** (Drizzle/Prisma throw via `FORWARDED_QB_METHODS`). **ADDITIVE projection** (appends after structured/raw selects or the default `*` — deliberately NOT `selectRaw`'s REPLACE semantics, so rows hydrate as full models with the alias as an extra attribute). `state.windows` compiles in `compileSelectBody` next to the aggregate subselects; bind-free (identifiers quoted, function name from the closed `WINDOW_FUNCTION_SQL` map = the injection gate, directions validated to asc/desc at the builder). No dialect seam — identical syntax on sqlite ≥3.25/pg/mysql 8. `orderBy` opts entries are `string | { column, direction }` (deliberately NOT `[col, dir]` tuples — flat two-string arrays are ambiguous with "two columns"). Aggregates-OVER/lag/lead/frames = documented `selectRaw` recipe; window-in-WHERE is invalid SQL → CTE/subquery. Methods on `HydratingQueryBuilder` + `Model.selectWindow` static — NOT the contract, zero QB-stub churn. Tests: `database/src/native/window.test.ts` (compiler pins + sqlite E2E + gated live-pg/mysql) + `orm/src/native/window.test.ts` (Model layer + guard).
- **`whereExists` / `whereNotExists` / `orWhere*` forms** — arbitrary `[NOT] EXISTS (…)` subqueries; **NATIVE engine only** (Drizzle/Prisma throw via `FORWARDED_QB_METHODS`). Body = another native query (proxy-unwrapped, same symbol as `union()`/CTEs — correlate via qualified `whereColumn('orders.userId', 'users.id')`, both sides identifier-quoted) or a raw SQL string + bindings. New `'exists'` `ConditionNode` kind rides `compileNodes` — the subquery compiles through the shared **`compileSubqueryBody`** (also used by CTE bodies: builder state + its unions, ORDER/LIMIT dropped; raw `?` rebinds) so its bindings sit exactly where the predicate sits in the WHERE. Composes inside groups/`whereNot`. For relation-shaped checks prefer `whereHas` (this is the no-declared-relation escape hatch). Builder body + bindings arg throws. Methods on `HydratingQueryBuilder` + `Model` statics (AND forms) — NOT the contract. Tests: `database/src/native/where-exists.test.ts` + `orm/src/native/where-exists.test.ts`.
- **`insertUsing(columns, query, bindings?)`** — `INSERT INTO t (cols) SELECT …`; **NATIVE engine only** (Drizzle/Prisma throw via `FORWARDED_QB_METHODS`). Subquery body = builder-or-raw via the shared `_subqueryBody()` resolver (same as `whereExists`/CTEs); `compileInsertUsing` quotes the REQUIRED column list (positional mapping — bare `INSERT INTO t SELECT` is a column-order footgun, so empty columns throw) + compiles the body through the shared `Bindings`. Count: `RETURNING *`→rows.length on sqlite/pg, `_affecting().affectedRows` on mysql (mirrors `upsert`). Bulk data-plane: NO observer events, NO fillable/guarded, NO keyType generation. **It's an async TERMINAL in the FORWARDED set — the proxy handler was fixed to pass non-`this` returns through (`out === target ? proxy : out`); previously it unconditionally returned the proxy and would have swallowed the Promise.** Gated live-mysql E2E covers the no-RETURNING count. Tests: `database/src/native/insert-using.test.ts` + `orm/src/native/insert-using.test.ts` (NOTE: async terminal guard asserts use `assert.rejects`, not `throws`).
- **Pessimistic locking (`lockForUpdate()` / `sharedLock()`)** — OPTIONAL on the `QueryBuilder` contract (capability-check before calling, like the native queue does: `typeof qb.lockForUpdate === 'function'`). **Native** (#837): `Dialect.lockSql()` → `FOR UPDATE`/`FOR SHARE` on pg/mysql, no-op on sqlite. **Drizzle**: real via the builder's `.for('update' | 'share')` (pg/mysql); NO-OP on sqlite (its select builders have no `.for()` and sqlite has no row locks — matches native); skipped on a union'd query (`FOR UPDATE` isn't valid SQL on a set operation). **Prisma THROWS** with a raw-transaction pointer (`DB.transaction(() => DB.select('… FOR UPDATE', binds))`) — a silent no-op would be a correctness bug for job-queue-style reservations. Only meaningful inside a `transaction()`. Tests: `orm-drizzle/lock.test.ts` (recorded dialect branching + sqlite E2E), `orm-prisma/lock.test.ts`.
- **Soft deletes**: `delete()` sets `deletedAt`; use `forceDelete()` for real removal
- **Pruning**: Models defining `static prunable()` are picked up by `pnpm rudder model:prune`. Default `pruneMode = 'instance'` re-queries each chunk and calls `instance.delete()` per row — soft-deletes apply, `deleting`/`deleted` observers fire, optional `static pruning(model)` runs first. `pruneMode = 'mass'` (`MassPrunable`) runs a single `qb.deleteAll()` per chunk — no observers, no hooks, soft-deletes bypassed (mirrors the existing bulk-delete primitive). Discovery uses `ModelRegistry.all()`. Runner is `pruneModels(opts)` exported from `@rudderjs/orm`; CLI subpath `@rudderjs/orm/commands/prune` registers the `model:prune` command. Re-queries (instead of `offset()` paging) because each delete shifts the offset.
- **Hydration**: every read path (`find`/`first`/`all`/`paginate`/`where(...).get()`/etc.) returns Model instances, not plain records. Instance methods (`save`/`fill`/`refresh`/`delete`/`replicate`/`is`/`isNot`/`trashed`/`related`) work directly on those instances. Use `Model.hydrate(record)` to wrap an external plain record (cached JSON, fixture). Adapters still return plain records — the hydrating QueryBuilder Proxy wraps them at the Model boundary.
- **Relations are deliberately thin (with one exception)**: `static relations` + `instance.related(name)` return a chainable QueryBuilder pre-filtered to the parent record. **Whole-row eager loading** is adapter-driven on Prisma (native `include`) but Model-layer-driven on adapters that set `eagerLoadStrategy: 'model-layer'` (Drizzle AND the native engine): the partition in `polymorphic-eager-load.ts` routes direct relations (`hasOne`/`hasMany`/`belongsTo`/`belongsToMany`) into `direct-eager-load.ts`'s `attachDirectRelations` (one batched WHERE-IN per relation, stitched onto parents — same machinery polymorphic relations use), so `Model.with(...)` works on Drizzle without a schema relation graph and on native without join SQL (the native QB's own `with()` stays a dev-warn no-op, reachable only via the `withWhereHas` constrained-eager fallback). Polymorphic relations are always Model-layer regardless. Undeclared/nested (`'a.b'`) names throw a clear error. **Aggregate eager loading IS shimmed** — `withCount` / `withSum` / `withMin` / `withMax` / `withAvg` / `withExists` on the QueryBuilder + the per-instance `loadCount` / `loadSum` / `loadMin` / `loadMax` / `loadAvg` / `loadExists` / `loadMissing` — because adapter divergence is too wide to push to userland (Prisma exposes `_count.select` natively but no portable per-relation sum; Drizzle has neither). The Model layer normalizes all overloads into `AggregateRequest[]` and dispatches to the adapter's `withAggregate(reqs)` (multi-row) or `_aggregate(fn, column?)` (single-scalar) terminal. Aggregate columns are stamped under deterministic camelCase aliases (`postsCount` / `postsSumViews`) and tagged on a `Symbol.for('rudderjs.orm.aggregates')` Set so `_toData()` excludes them on writes. `withCount` on `belongsTo` and on `morphTo` throws with helpful messages. Relation types: `hasOne`, `hasMany`, `belongsTo`, `belongsToMany`, `hasOneThrough`, `hasManyThrough`, `morphMany`, `morphOne`, `morphTo`, `morphToMany`, `morphedByMany`. **Through relations** (`hasOneThrough`/`hasManyThrough`, `src/relations/has-through.ts`) reach a distant relation via an intermediate model (`Country → hasManyThrough(Post, User)`: `countries.id=users.countryId` then `users.id=posts.userId`); declared `{ type, model: () => Related, through: () => Through, firstKey?, secondKey?, localKey?, secondLocalKey? }` with Laravel-convention defaults (`firstKey`=`${camelCase(Parent)}Id`, `secondKey`=`${camelCase(Through)}Id`, local keys = each PK). Both lazy `related()` (deferred-proxy in `pivot-deferred.ts` → `hasThroughDeferredQb`, two `WHERE IN` hops) and eager `with()` (`attachHasThrough`) resolve in the Model layer with NO join SQL / no adapter change. **Always** routed Model-layer regardless of adapter eager strategy (`isThrough` in the partition pushes them to the `direct` lane — no adapter can express the two-hop walk natively). `whereHas`/`withCount` on a through relation **throw** a clear "not supported yet" error (two-level EXISTS/aggregate deferred). NB: TS can't drop a two-literal discriminant union member via a `||`-guard + return, so the `hasOne|hasMany` fallthroughs in `index.ts`/`where-has.ts`/`aggregate.ts` positively narrow via `def as Extract<RelationDefinition, { type: 'hasOne'|'hasMany'|'belongsTo' }>` (the `Extract` filter MUST list all three literals of that shared member or it resolves to `never`). Polymorphic columns use **camelCase** (`commentableId` / `commentableType`) for ORM consistency — a deliberate divergence from Laravel's snake_case. The discriminator stored in `{morphName}Type` defaults to the parent's class name; override with `static morphAlias = 'post'` for rename-safe storage (treat as immutable once data exists). `morphTo` requires a closed `types: () => [...]` list; lookup matches `morphAlias ?? Class.name`. Dev-mode collision guard (`NODE_ENV !== 'production'`) checks the `types` list for duplicate discriminators on first resolution. `morphToMany` / `morphedByMany` route reads/writes through a shared pivot table carrying `{morphName}Id` + `{morphName}Type`; the owning side (`morphToMany`) filters/stamps its own discriminator, the inverse (`morphedByMany`) filters by the related class's. Each `morphedByMany` declaration targets **one** concrete inverse class — declare one relation per inverse target (`Tag.posts`, `Tag.videos`) rather than walking a polymorphic types list. The `model: () => Class` thunk is mandatory to dodge circular imports.
- **`whereHas` / `whereDoesntHave` / `withWhereHas` / `whereBelongsTo`**: relation predicate filters routed through `QueryBuilder.whereRelationExists(predicate)`. The Model layer builds a `RelationExistencePredicate` (related table, parent/related columns, constraint wheres, optional `extraEquals` for morph discriminators, optional `through` for pivots) and adapters translate to native shape. Constrain callbacks run against a recording-only QueryBuilder that captures `.where()` calls into a flat `WhereClause[]` and treats every other chainable method as a no-op. Nested `whereHas` inside the constrain callback throws — recursive predicates are deferred to v2. **`morphTo` is intentionally not supported** — the related table is dynamic so a single subquery can't represent it; filter on `{morphName}Id` / `{morphName}Type` directly. `withWhereHas` uses the adapter's optional `withConstrained(relation, wheres)` when present (Prisma → nested `include: { rel: { where } }`); otherwise it falls back to plain `with(relation)`. On the **Drizzle adapter** plain `Model.with(relation)` now works (Model-layer batched loader, see the relations bullet above), but the adapter's QB-level `with()` still **throws** — and that's the path `withWhereHas`'s constrained-eager fallback hits (`q.with(rel)`), so `withWhereHas` still throws on Drizzle. For constrained eager loading on Drizzle, use `whereHas(relation)` for the filter (it never calls `with()`) and load the constrained children explicitly via the `related()` accessor. The Drizzle adapter requires the related table (and pivot, when present) to be registered via `tables: { ... }` config or `DrizzleTableRegistry.register(name, table)` — error surfaces with the missing table name. Soft-delete scoping inside the relation predicate is the constrain callback's responsibility; document `q.where('deletedAt', null)` as the explicit pattern. **OR/count operators** (`orWhereHas`/`orWhereDoesntHave`/`has(rel,op,n)`/`orHas`): the `RelationExistencePredicate` carries optional `boolean: 'AND'|'OR'` (OR-rooting) + `count: {operator,value}` (count comparison → `(SELECT COUNT(*) …) op n`). NATIVE only — the compiler folds OR predicates into one group AND-wrapped by the soft-delete scope (the all-AND text path is preserved byte-for-byte so existing SQL holds), and emits the COUNT form when `count` is set (integer inlined via `asInt`, not bound). **Drizzle + Prisma THROW** for the count/OR forms (no structured equivalent) and point at `whereHas`/`DB.select`; plain `whereHas`/`whereDoesntHave` keep working on every adapter. Tests: `native/where-has-ops.test.js`.
- **`belongsToMany`** ships pivot mutations on a per-relation accessor: `attach(ids | {id: pivotData}, flatPivot?)`, `detach(ids?)`, `sync(ids, flatPivot?)`. The accessor auto-installs on the parent prototype on first query (`user.roles().attach(...)`); call `Model.belongsToMany(parent, name)` directly for typed wrappers. Reads via `parent.related('roles')` use a deferred QueryBuilder Proxy — chains stay sync, the pivot lookup runs on terminal evaluation, and mutation methods (`create`/`update`/`delete`/`insertMany`/`deleteAll`) throw with a helpful pointer. v1 deliberately does not surface pivot columns on reads or auto-stamp pivot timestamps; both are gated on real demand. Adapter contract: `QueryBuilder.insertMany(rows)` and `deleteAll()` (returns count) are the generic primitives the M2M layer composes — every adapter implements them. **`morphToMany` / `morphedByMany`** share the same accessor surface (`MorphToManyAccessor` / `MorphedByManyAccessor`) and the same Proxy-based deferred-read pipeline; the only delta is one extra `where({morphName}Type, value)` on every pivot query and one extra column written on every pivot insert. Don't declare a class field `tags!: () => ...` to type the auto-installed accessor — it shadows the prototype method. Type the explicit override (`tags() { return Model.morphToMany(this, 'tags') }`) instead.
- **Route model binding**: `static routeKey` (defaults to `'id'`) + `static findForRoute(value)` are the duck-typed surface that `@rudderjs/router`'s `router.bind(name, ModelClass)` consumes. The router doesn't depend on `@rudderjs/orm` — anything with these statics works. Default `findForRoute` runs `Model.where(routeKey, value).first()`; subclasses override the static signature as `Promise<Model | null>` (looser than `InstanceType<T>` so subclass overrides type-check under `exactOptionalPropertyTypes`).
- **Internal fields are `#`-private**: serialization overrides (`#instanceHidden`/`#instanceVisible`) use ECMAScript private syntax so they never appear in `Object.entries`, object spread, or `JSON.stringify` — keeps hydrated instances wire-format clean for Prisma writes and Telescope serialization.
- **`_toData()` filters `undefined`**: a class field declared `id!: number` becomes an enumerable `undefined` own property at construction time; `save()`/`replicate()` drop those so Prisma never sees `id: undefined` on inserts.

## Conventions

- Table name defaults to lowercase class name + `'s'` (User → users)
- Primary key defaults to `'id'`
- `@Hidden`, `@Visible`, `@Appends`, `@Cast` decorators configure serialization
- Only depends on `@rudderjs/contracts` — no runtime DB driver dependency
