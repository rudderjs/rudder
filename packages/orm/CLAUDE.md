# @rudderjs/orm

Eloquent-style ORM — Model base class, casts, attributes, scopes, observers, factories, and API resources.

## Key Files

- `src/index.ts` — `Model` base class (query, CRUD, scopes, observers, serialization), decorators, `ModelRegistry`, `ModelNotFoundError`
- `src/cast.ts` — Built-in casts (string, integer, float, boolean, date, json, encrypted) + custom `CastUsing` + `vector({ dimensions })` factory (#B7 Phase 1) — returns a `CastUsing` class that serializes `number[]` ⇄ pgvector text format with write-time dimension/finiteness validation
- `src/vector-errors.ts` — `VectorDimensionMismatchError`, `VectorStorageUnsupportedError`, `MissingEmbedderError` (all carry stable `code` fields for app `instanceof` + `.code` dispatch)
- `src/attribute.ts` — `Attribute.make({ get, set })` for computed accessors/mutators
- `src/collection.ts` — `ModelCollection` typed array wrapper with ORM operations
- `src/factory.ts` — `ModelFactory` for testing with states and sequences
- `src/resource.ts` — `JsonResource` / `ResourceCollection` for API response transformation
- `src/seeder.ts` — `Seeder` base class for `db:seed` runner
- `src/commands/make-factory.ts` — `makeFactorySpec` MakeSpec for `pnpm rudder make:factory <Name>` (writes `app/Factories/<Name>Factory.ts`)
- `src/commands/make-seeder.ts` — `makeSeederSpec` MakeSpec for `pnpm rudder make:seeder <Name>` (writes `database/seeders/<Name>Seeder.ts`)

## Architecture Rules

- **Adapter pattern**: `ModelRegistry.set(adapter)` plugs in the actual DB driver (Prisma adapter lives in playground)
- **Event lifecycle**: `Model.create/update/delete()` trigger observer events; `query().create()` bypasses them. Events: `retrieved`, `creating`/`created`, `updating`/`updated`, `saving`/`saved` (fire on both create+update), `deleting`/`deleted`, `restoring`/`restored`. `Model.withoutEvents(fn)` mutes per-class. **`increment`/`decrement` deliberately do NOT fire observer events** — they're a pure data-plane operation; the observer payload would have to be either the delta or the resolved value, both of which conflict with atomic SQL semantics.
- **Serialization order**: casts (get) → attributes (getter) → appends → visible/hidden filtering → `toJSON()`
- **Mass assignment**: `static fillable` (allowlist) and `static guarded` (denylist; `['*']` locks all) are enforced on `Model.create()`, `Model.update()`, and `instance.fill()` — keys outside the policy are silently dropped. Both default to `[]` (no enforcement, back-compat). `fillable` wins when both are set. Bypass paths: `instance.forceFill(data)` and direct property assignment + `instance.save()` (data set property-by-property is not mass-assignment, so `save()` skips the filter via internal `_doCreate`/`_doUpdate`). **`Model.upsert()` and `insertMany` also bypass the filter** (bulk ops) — write-side casts/mutators still apply on upsert.
- **`chunk(size, cb)` / `lazy(size?)`** — memory-bounded iteration, implemented in the `_hydratingQb` proxy (Model-layer only; NO contracts/adapter change — they page via the existing `limit`/`offset`/`get` primitives, mutating `target`'s limit/offset each pass and reusing the `get` hydration path). `chunk` re-queries per page (offset paging), returns `true`/`false` (callback `false` bails); `lazy` is an async generator (default page 1000). Both also exposed as Model statics. Offset-paging relies on a consistent sort — recommend an `orderBy` (mirrors Laravel `chunk`, which also doesn't auto-order). Typed on `HydratingQueryBuilder` (NOT the `QueryBuilder` contract). Tests in `chunk-lazy.test.js` (own file in the orm test-list).
- **`Model.upsert(rows, uniqueBy, update?)`** — bulk insert-or-update (Laravel parity). Model layer normalizes `uniqueBy`→array, computes the default `update` set (every inserted column minus `uniqueBy`), applies write-time casts/mutators per row, then dispatches to the adapter's optional `QueryBuilder.upsert(rows, uniqueBy[], update[])`. Native + Drizzle emit ONE statement (`ON CONFLICT … DO UPDATE`/`DO NOTHING` on sqlite/pg via the new `Dialect.upsertClause()` seam + `compileInsert({upsert})`; `ON DUPLICATE KEY UPDATE` on MySQL — `uniqueBy` ignored there since MySQL keys off any unique index). Prisma has no portable bulk ON CONFLICT → per-row `delegate.upsert` batched in one `$transaction`. **No observer events fire** (pure data-plane, like `increment`/`insertMany`). Empty `update` = insert-or-ignore. `upsert?` is OPTIONAL on the `OrmAdapter`/`QueryBuilder` contract — the Model layer throws an adapter-named error if absent. MySQL's returned count is rows-*touched* (1/insert, 2/update), not distinct.
- **Raw expressions (`selectRaw`/`whereRaw`/`orWhereRaw`/`orderByRaw` + `raw(...)`/`DB.raw(...)`)** — raw-SQL escape hatch on the `QueryBuilder` contract (required methods, not `HydratingQueryBuilder`-only). The `Expression` wrapper + `raw()` live in **`@rudderjs/contracts`** (re-exported by `@rudderjs/database` so `DB.raw()` keeps its path) — NOT in `@rudderjs/database`, because the QB raw methods are Model/client-reachable and `@rudderjs/database` is node-only. Raw fragments use `?` placeholders + a positional `bindings` array; the native compiler's `compileRaw` rebinds each `?` to the dialect placeholder (`$n` on pg) through the **one shared positional `Bindings`** — so select-list raw bindings land before WHERE's, which land before ORDER BY's (SQL text order). `?` count must equal `bindings.length` or it throws. `selectRaw` REPLACES the default `*` projection (Laravel semantics). A `raw()` value in a `where(col, op, raw(...))` splices verbatim (no bind); `orderBy(raw(...))` is a raw order item. **Native `_orders` is `OrderItem[]` (structured `OrderClause` | `{kind:'raw'}`) and Drizzle's `_orders` is likewise a union — one list keeps interleaved structured+raw order.** Adapter divergence: **native** = full; **Drizzle** = whereRaw/orWhereRaw/orderByRaw via its `sql` template (the module-level `rawToSql` splits on `?` and parameterizes), `selectRaw` THROWS (typed select can't map a raw projection back to a model); **Prisma** = all four THROW (structured client) pointing at `DB.select(sql, bindings)`. Tests in `native/raw-expr.test.ts` (own line in orm test-list) + `orm-drizzle/raw-expr.test.ts` + `orm-prisma/raw-expr.test.ts`. **Adding a required QB method breaks ~15 hand-rolled QB test stubs (orm + ai) — they each need the new methods.**
- **`whereX` query sugar** (`whereIn`/`whereNotIn`/`whereNull`/`whereNotNull`/`whereBetween`/`whereNotBetween` + `orWhere*` forms, `when`/`unless`, `latest`/`oldest`, and the scalar terminals `pluck`/`value`/`sum`/`max`/`min`/`avg`/`exists`/`doesntExist`) — implemented ENTIRELY in the `_hydratingQb` proxy (Model-layer; like `chunk`/`lazy`), composing the existing `where`/`orWhere`/`whereGroup`/`orderBy`/`get`/`first`/`_aggregate` primitives. **NO contracts/adapter change → all 3 adapters get them free, and no QB-stub churn.** `whereBetween`→grouped `>=a AND <=b`; `whereNotBetween`→grouped `<a OR >b`; `whereNull`→`where(col,'=',null)` (compiler emits IS NULL). `when(v,cb,else?)`/`unless` pass `(proxy, value)` to the callback. `pluck`/`value` hydrate via `wrapMany`/`wrapMaybe` so casts apply; `sum`/`max`/`min`/`avg`/`exists` route through the QB `_aggregate(fn, col?)` terminal (`exists` returns boolean). Typed on `HydratingQueryBuilder` (NOT the `QueryBuilder` contract); each also a `Model` static entry point. Tests in `where-sugar.test.js` (own orm test-list line). Deferred to a later PR (need real SQL / contract change): `whereNot` group-negation, date helpers, structured `select()`/`distinct()`.
- **`whereColumn` / `orWhereColumn`** (`whereColumn('a','b')` / `whereColumn('a','>','b')`) — compare two columns, both sides identifier-quoted per dialect (the reason it can't ride on `whereRaw`, which is verbatim). NOT pure Model-layer sugar — needs real SQL: native adds a `'column'` `ConditionNode` kind compiled as `quoteId(left) op quoteId(right)`; **Drizzle** renders both as column refs through `sql` (quoted per dialect); **Prisma** throws → `DB.select`/`whereRaw` (no column-vs-column in its query API). Methods live on `HydratingQueryBuilder` (required) + `Model` statics — NOT on the `QueryBuilder` contract, so **zero QB-stub churn**. Tests: `native/where-column.test.js` (own orm test-list line) + drizzle/prisma.
- **Application-generated primary keys (`static keyType`)**: `'int'` (default) = database auto-increment, ORM never sets the PK. `'uuid'` / `'ulid'` = the Model layer stamps a fresh key on `_doCreate` (covers `create`/`save`/`firstOrCreate`/`updateOrCreate`) when the PK is unset — UUID v4 via Web Crypto `randomUUID`, ULID via a 26-char Crockford Base32 generator (`src/keys.ts`, 48-bit ms timestamp prefix → lexicographically sortable). Both use `globalThis.crypto` so `keys.ts` stays client-bundle safe (no `node:` import). Pure Model-layer (in `_ensureGeneratedKey`, called from `_doCreate`) → all 3 adapters free, no contract change. A caller-supplied key is never overwritten. Bulk `insertMany`/`upsert` are NOT covered (pure data-plane, like their observer-skipping). Tests: `key-type.test.js` (own orm test-list line).
- **Joins + structured `select()`** (`join`/`leftJoin`/`rightJoin`/`crossJoin` + `select(...columns)`) — **NATIVE engine only**; **Drizzle + Prisma THROW** with a pointer to the native engine / `DB.select(...)` (same result-shape-mapping reason `selectRaw` throws — a join/projection changes the row shape the typed clients map back to one model). Methods live on `HydratingQueryBuilder` + `Model` statics, **NOT the `QueryBuilder` contract → zero QB-stub churn** (the proxy's `default` case auto-forwards them to the adapter QB and re-wraps `this`→`proxy`). Native: `NativeQueryState` gains `joins?: JoinNode[]` + `selects?: string[]`; `compileJoins` emits `[INNER|LEFT|RIGHT|CROSS] JOIN "t" ON …` **between FROM and WHERE** so a join `where`-value binds in that slot (after SELECT-list bindings, before WHERE's — shared positional `Bindings`); `compileCount` includes joins too (fan-out). `JoinClause` (the callback sub-builder for compound ON) is a **standalone interface in `@rudderjs/contracts`** (type-only, NOT a `QueryBuilder` member) — `on`/`orOn` = column-vs-column (quoted, nothing binds), `where`/`orWhere` = column-vs-value (binds). `select(...)` quotes qualified `table.col` and REPLACES `*` (combines with `selectRaw`: structured first, then raw). Join `join('t','a','=','b')` simple form OR `join('t', j => j.on(...).where(...))`; the bare two-column shorthand (`a = b` with no operator) is only typed via `j.on('a','b')`. RIGHT JOIN needs SQLite 3.39+. Tests: `native/joins.test.js` (own orm test-list line; compiler units + sqlite E2E) + throw assertions appended to `orm-drizzle`/`orm-prisma` `raw-expr.test.ts`. **Closes the structured `select()`/`distinct()` deferral noted in the whereX bullet (select done; distinct still pending).** Follow-ups (separate PRs): `union`/`unionAll`, real Drizzle joins, `distinct()`.
- **`groupBy` / `having`** (`groupBy(...cols)` + `having(col,op,val)`/`orHaving`/`havingRaw(sql,binds)`/`orHavingRaw`) — **NATIVE engine only**; **Drizzle + Prisma THROW** (same `_builderUnsupported` pointer as joins). `NativeQueryState` gains `groupBy?: string[]` + `having?: HavingNode[]` (`HavingNode` = `clause` (binds) | `raw`). Compiler emits `GROUP BY` after WHERE and `HAVING` after GROUP BY (before ORDER BY) — HAVING values bind after WHERE's, before ORDER BY's (shared positional `Bindings`). **`compileCount` wraps a grouped query** — `SELECT COUNT(*) FROM (SELECT 1 … GROUP BY … HAVING …) AS aggregate` — so `count()`/`paginate()` return the number of GROUPS (Laravel parity), not per-group row counts; the no-GROUP-BY path stays a plain scalar `COUNT(*)`. `having('alias',op,v)` works on a SELECT alias on SQLite/MySQL but NOT Postgres (use `havingRaw('COUNT(*) op ?',[n])` for portability). Methods on `HydratingQueryBuilder` + `Model` statics (`groupBy`/`having`/`havingRaw`), NOT the contract → zero stub churn. Tests: `native/group-having.test.js` (own orm test-list line).
- **`union` / `unionAll`** (`base.union(other)` / `base.unionAll(other)`) — **NATIVE engine only**; **Drizzle + Prisma THROW** (`_builderUnsupported`). `NativeQueryState` gains `unions?: {all,state}[]`. `compileSelect` was refactored to extract **`compileSelectBody`** (projection→HAVING, no ORDER/LIMIT/lock) — the base body + each member body are joined with `UNION [ALL]` over the **one shared positional `Bindings`** (base binds first, then members in order), then the **base query's** ORDER BY / LIMIT / OFFSET / lock apply to the whole result (member ORDER/LIMIT are dropped). `compileCount` wraps the union (`SELECT COUNT(*) FROM (… UNION …) AS aggregate`) so count()/paginate() count combined rows (takes precedence over the GROUP BY wrap). The no-union path is byte-identical to before (regression-safe). **Unwrap mechanism:** `union(other)` receives the `HydratingQueryBuilder` Proxy and reads its underlying native builder via `Symbol.for('rudderjs.orm.qb.target')` — the proxy answers that symbol with its `target`; defined independently (NOT imported) in both `index.ts` and `query-builder.ts` so the node-only native module stays out of the client-reachable import graph. Members must be native builders (throws otherwise). Methods on `HydratingQueryBuilder` only (no Model static — union starts from a base query). Tests: `native/union.test.js` (own orm test-list line).
- **Soft deletes**: `delete()` sets `deletedAt`; use `forceDelete()` for real removal
- **Pruning**: Models defining `static prunable()` are picked up by `pnpm rudder model:prune`. Default `pruneMode = 'instance'` re-queries each chunk and calls `instance.delete()` per row — soft-deletes apply, `deleting`/`deleted` observers fire, optional `static pruning(model)` runs first. `pruneMode = 'mass'` (`MassPrunable`) runs a single `qb.deleteAll()` per chunk — no observers, no hooks, soft-deletes bypassed (mirrors the existing bulk-delete primitive). Discovery uses `ModelRegistry.all()`. Runner is `pruneModels(opts)` exported from `@rudderjs/orm`; CLI subpath `@rudderjs/orm/commands/prune` registers the `model:prune` command. Re-queries (instead of `offset()` paging) because each delete shifts the offset.
- **Hydration**: every read path (`find`/`first`/`all`/`paginate`/`where(...).get()`/etc.) returns Model instances, not plain records. Instance methods (`save`/`fill`/`refresh`/`delete`/`replicate`/`is`/`isNot`/`trashed`/`related`) work directly on those instances. Use `Model.hydrate(record)` to wrap an external plain record (cached JSON, fixture). Adapters still return plain records — the hydrating QueryBuilder Proxy wraps them at the Model boundary.
- **Relations are deliberately thin (with one exception)**: `static relations` + `instance.related(name)` return a chainable QueryBuilder pre-filtered to the parent record. **Whole-row eager loading** is adapter-driven on Prisma (native `include`) but Model-layer-driven on adapters that set `eagerLoadStrategy: 'model-layer'` (Drizzle): the partition in `polymorphic-eager-load.ts` routes direct relations (`hasOne`/`hasMany`/`belongsTo`/`belongsToMany`) into `direct-eager-load.ts`'s `attachDirectRelations` (one batched WHERE-IN per relation, stitched onto parents — same machinery polymorphic relations use), so `Model.with(...)` works on Drizzle without a schema relation graph. Polymorphic relations are always Model-layer regardless. Undeclared/nested (`'a.b'`) names throw a clear error. **Aggregate eager loading IS shimmed** — `withCount` / `withSum` / `withMin` / `withMax` / `withAvg` / `withExists` on the QueryBuilder + the per-instance `loadCount` / `loadSum` / `loadMin` / `loadMax` / `loadAvg` / `loadExists` / `loadMissing` — because adapter divergence is too wide to push to userland (Prisma exposes `_count.select` natively but no portable per-relation sum; Drizzle has neither). The Model layer normalizes all overloads into `AggregateRequest[]` and dispatches to the adapter's `withAggregate(reqs)` (multi-row) or `_aggregate(fn, column?)` (single-scalar) terminal. Aggregate columns are stamped under deterministic camelCase aliases (`postsCount` / `postsSumViews`) and tagged on a `Symbol.for('rudderjs.orm.aggregates')` Set so `_toData()` excludes them on writes. `withCount` on `belongsTo` and on `morphTo` throws with helpful messages. Relation types: `hasOne`, `hasMany`, `belongsTo`, `belongsToMany`, `hasOneThrough`, `hasManyThrough`, `morphMany`, `morphOne`, `morphTo`, `morphToMany`, `morphedByMany`. **Through relations** (`hasOneThrough`/`hasManyThrough`, `src/relations/has-through.ts`) reach a distant relation via an intermediate model (`Country → hasManyThrough(Post, User)`: `countries.id=users.countryId` then `users.id=posts.userId`); declared `{ type, model: () => Related, through: () => Through, firstKey?, secondKey?, localKey?, secondLocalKey? }` with Laravel-convention defaults (`firstKey`=`${camelCase(Parent)}Id`, `secondKey`=`${camelCase(Through)}Id`, local keys = each PK). Both lazy `related()` (deferred-proxy in `pivot-deferred.ts` → `hasThroughDeferredQb`, two `WHERE IN` hops) and eager `with()` (`attachHasThrough`) resolve in the Model layer with NO join SQL / no adapter change. **Always** routed Model-layer regardless of adapter eager strategy (`isThrough` in the partition pushes them to the `direct` lane — no adapter can express the two-hop walk natively). `whereHas`/`withCount` on a through relation **throw** a clear "not supported yet" error (two-level EXISTS/aggregate deferred). NB: TS can't drop a two-literal discriminant union member via a `||`-guard + return, so the `hasOne|hasMany` fallthroughs in `index.ts`/`where-has.ts`/`aggregate.ts` positively narrow via `def as Extract<RelationDefinition, { type: 'hasOne'|'hasMany'|'belongsTo' }>` (the `Extract` filter MUST list all three literals of that shared member or it resolves to `never`). Polymorphic columns use **camelCase** (`commentableId` / `commentableType`) for ORM consistency — a deliberate divergence from Laravel's snake_case. The discriminator stored in `{morphName}Type` defaults to the parent's class name; override with `static morphAlias = 'post'` for rename-safe storage (treat as immutable once data exists). `morphTo` requires a closed `types: () => [...]` list; lookup matches `morphAlias ?? Class.name`. Dev-mode collision guard (`NODE_ENV !== 'production'`) checks the `types` list for duplicate discriminators on first resolution. `morphToMany` / `morphedByMany` route reads/writes through a shared pivot table carrying `{morphName}Id` + `{morphName}Type`; the owning side (`morphToMany`) filters/stamps its own discriminator, the inverse (`morphedByMany`) filters by the related class's. Each `morphedByMany` declaration targets **one** concrete inverse class — declare one relation per inverse target (`Tag.posts`, `Tag.videos`) rather than walking a polymorphic types list. The `model: () => Class` thunk is mandatory to dodge circular imports.
- **`whereHas` / `whereDoesntHave` / `withWhereHas` / `whereBelongsTo`**: relation predicate filters routed through `QueryBuilder.whereRelationExists(predicate)`. The Model layer builds a `RelationExistencePredicate` (related table, parent/related columns, constraint wheres, optional `extraEquals` for morph discriminators, optional `through` for pivots) and adapters translate to native shape. Constrain callbacks run against a recording-only QueryBuilder that captures `.where()` calls into a flat `WhereClause[]` and treats every other chainable method as a no-op. Nested `whereHas` inside the constrain callback throws — recursive predicates are deferred to v2. **`morphTo` is intentionally not supported** — the related table is dynamic so a single subquery can't represent it; filter on `{morphName}Id` / `{morphName}Type` directly. `withWhereHas` uses the adapter's optional `withConstrained(relation, wheres)` when present (Prisma → nested `include: { rel: { where } }`); otherwise it falls back to plain `with(relation)`. On the **Drizzle adapter** plain `Model.with(relation)` now works (Model-layer batched loader, see the relations bullet above), but the adapter's QB-level `with()` still **throws** — and that's the path `withWhereHas`'s constrained-eager fallback hits (`q.with(rel)`), so `withWhereHas` still throws on Drizzle. For constrained eager loading on Drizzle, use `whereHas(relation)` for the filter (it never calls `with()`) and load the constrained children explicitly via the `related()` accessor. The Drizzle adapter requires the related table (and pivot, when present) to be registered via `tables: { ... }` config or `DrizzleTableRegistry.register(name, table)` — error surfaces with the missing table name. Soft-delete scoping inside the relation predicate is the constrain callback's responsibility; document `q.where('deletedAt', null)` as the explicit pattern. **OR/count operators** (`orWhereHas`/`orWhereDoesntHave`/`has(rel,op,n)`/`orHas`): the `RelationExistencePredicate` carries optional `boolean: 'AND'|'OR'` (OR-rooting) + `count: {operator,value}` (count comparison → `(SELECT COUNT(*) …) op n`). NATIVE only — the compiler folds OR predicates into one group AND-wrapped by the soft-delete scope (the all-AND text path is preserved byte-for-byte so existing SQL holds), and emits the COUNT form when `count` is set (integer inlined via `asInt`, not bound). **Drizzle + Prisma THROW** for the count/OR forms (no structured equivalent) and point at `whereHas`/`DB.select`; plain `whereHas`/`whereDoesntHave` keep working on every adapter. Tests: `native/where-has-ops.test.js`.
- **`belongsToMany`** ships pivot mutations on a per-relation accessor: `attach(ids | {id: pivotData}, flatPivot?)`, `detach(ids?)`, `sync(ids, flatPivot?)`. The accessor auto-installs on the parent prototype on first query (`user.roles().attach(...)`); call `Model.belongsToMany(parent, name)` directly for typed wrappers. Reads via `parent.related('roles')` use a deferred QueryBuilder Proxy — chains stay sync, the pivot lookup runs on terminal evaluation, and mutation methods (`create`/`update`/`delete`/`insertMany`/`deleteAll`) throw with a helpful pointer. v1 deliberately does not surface pivot columns on reads or auto-stamp pivot timestamps; both are gated on real demand. Adapter contract: `QueryBuilder.insertMany(rows)` and `deleteAll()` (returns count) are the generic primitives the M2M layer composes — every adapter implements them. **`morphToMany` / `morphedByMany`** share the same accessor surface (`MorphToManyAccessor` / `MorphedByManyAccessor`) and the same Proxy-based deferred-read pipeline; the only delta is one extra `where({morphName}Type, value)` on every pivot query and one extra column written on every pivot insert. Don't declare a class field `tags!: () => ...` to type the auto-installed accessor — it shadows the prototype method. Type the explicit override (`tags() { return Model.morphToMany(this, 'tags') }`) instead.
- **Route model binding**: `static routeKey` (defaults to `'id'`) + `static findForRoute(value)` are the duck-typed surface that `@rudderjs/router`'s `router.bind(name, ModelClass)` consumes. The router doesn't depend on `@rudderjs/orm` — anything with these statics works. Default `findForRoute` runs `Model.where(routeKey, value).first()`; subclasses override the static signature as `Promise<Model | null>` (looser than `InstanceType<T>` so subclass overrides type-check under `exactOptionalPropertyTypes`).
- **Internal fields are `#`-private**: serialization overrides (`#instanceHidden`/`#instanceVisible`) use ECMAScript private syntax so they never appear in `Object.entries`, object spread, or `JSON.stringify` — keeps hydrated instances wire-format clean for Prisma writes and Telescope serialization.
- **`_toData()` filters `undefined`**: a class field declared `id!: number` becomes an enumerable `undefined` own property at construction time; `save()`/`replicate()` drop those so Prisma never sees `id: undefined` on inserts.

## Conventions

- Table name defaults to lowercase class name + `'s'` (User → users)
- Primary key defaults to `'id'`
- `@Hidden`, `@Visible`, `@Appends`, `@Cast` decorators configure serialization
- Only depends on `@rudderjs/contracts` — no runtime DB driver dependency
